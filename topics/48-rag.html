RAG & Retrieval
Retrieval-Augmented Generation (RAG) is the dominant pattern for grounding LLM outputs in real, current, sourced knowledge — without fine-tuning. A RAG pipeline retrieves relevant documents from a knowledge base and injects them into the LLM's context at inference time. Getting every stage right — chunking, embedding, retrieval, reranking, generation — is the difference between a hallucinating chatbot and a reliable production system.
Why RAG Exists
  • LLMs hallucinate — parametric memory (weights) is unreliable for specific facts, recent events, proprietary knowledge
  • Knowledge cutoff — fine-tuning on new data is slow, expensive, and degrades general capabilities
  • No sourcing — base LLMs can't cite where a fact came from; RAG retrieves the source document
  • Privacy — sensitive company data can't go into model training; RAG keeps it in a controlled retrieval layer
  • Cost — retrieving and injecting 3–5 relevant passages is vastly cheaper than a 128K-token context

Common mistake

Fine-tuning to inject facts. Fine-tuning teaches the model how to behave, not what to know. Factual knowledge encoded in weights is unreliable and goes stale — always use RAG for knowledge.

Part 2 — Chunking Strategies
Chunking — The Most Underrated RAG Decision
Chunk size controls the granularity of retrieval. Too large: retrieved chunks contain irrelevant context, polluting the LLM. Too small: single chunks lack enough context to be useful.
StrategyHowBest forChunk size
Fixed-sizeSplit by token/char count with overlapBaseline; homogeneous text256–512 tokens, 20% overlap
Sentence / paragraphSplit on . / \n\nProse documents; natural boundaries1–5 sentences
Semantic chunkingSplit where embedding cosine similarity drops between adjacent sentencesMixed-topic documents; blog postsVariable; avg ~400 tokens
Recursive character splitterTry \n\n → \n → . → in order; respect structureGeneral purpose; LangChain default512–1024 tokens
Document-awareRespect markdown headers, PDF sections, code blocks as natural boundariesStructured docs, codebases, wikisSection-sized
Small-to-big / Parent-childEmbed small chunks; retrieve parent (larger) chunk for contextWhen context needed beyond single chunkChild: 128 tok; parent: 512 tok
Practical guide

Start with recursive splitter at 512 tokens / 20% overlap. Measure recall@5 on 50 question-answer pairs from your actual corpus. Switch to semantic chunking if recall is low (document topics shift abruptly). Use parent-child chunking if answers keep spanning chunk boundaries.

Part 3 — Retrieval Strategies
Hybrid Search — Dense + Sparse Combined
Dense retrieval (embedding similarity) handles semantic matching but misses exact keywords. Sparse retrieval (BM25) handles exact keywords but misses paraphrases. Hybrid search combines both — consistently outperforms either alone.
BM25 — the sparse baseline
$$\text{BM25}(q,d) = \sum_{i=1}^{n} \text{IDF}(q_i) \cdot \frac{f(q_i,d) \cdot (k_1+1)}{f(q_i,d) + k_1(1-b+b\cdot\frac{|d|}{\text{avgdl}})}$$
  • Term frequency weighted by inverse document frequency
  • Handles exact keyword match, product codes, names, jargon
  • Fast: inverted index lookup, no GPU needed
  • Fails on synonyms, paraphrases, semantic queries
Reciprocal Rank Fusion (RRF)
$$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + r(d)}, \quad k=60$$
  • Merge ranked lists from dense + sparse without needing score normalisation
  • k=60 is a robust default; larger k reduces impact of top-ranked items
  • Alternative: weighted linear combination of normalised scores (requires calibration)
  • Weaviate hybrid search uses RRF; Qdrant supports configurable fusion
BGE-M3: dense + sparse in one model

BGE-M3 generates dense embeddings, sparse (SPLADE-style) weights, and ColBERT late-interaction vectors from a single encoder. Eliminates the need for separate BM25 + dense pipelines — state-of-the-art on BEIR benchmark for hybrid retrieval.

Part 4 — Reranking
Reranking — Why Retrieval Recall ≠ Precision
ANN retrieval (bi-encoder) is fast but imprecise — it embeds query and document independently, losing cross-attention signal. A cross-encoder reranker reads query + document together via full attention, producing much more accurate relevance scores. The two-stage pipeline gives you the speed of ANN + the precision of cross-attention.
Cross-encoder reranker
  • Input: [CLS] query [SEP] document [SEP] → relevance score
  • Full cross-attention between query and document tokens
  • Too slow for the full corpus (O(n) forward passes); run on top-50–100 from ANN
  • Models: BGE reranker, Cohere rerank-3, ms-marco-MiniLM cross-encoders
  • Typically improves NDCG@10 by 5–15% over bi-encoder alone

💡 Practical Eval Setup

Build a golden dataset of 50–100 question / expected-answer / source-document triples from your actual corpus. Run RAGAS on every pipeline change. Track faithfulness first — it's the hallucination metric and the most important for production trust. Context precision and recall diagnose whether the problem is in retrieval or generation.

Part 6 — Production RAG Patterns
Agentic RAG & Multi-hop Reasoning
Simple RAG retrieves once. Real-world questions often require multiple retrieval steps — each answer informing the next query. Agentic RAG lets the LLM decide when and what to retrieve.
PatternMechanismUse case
Iterative RAGRetrieve → answer partial → retrieve again with refined query (loop until sufficient)Multi-hop questions: "Who founded the company that acquired X?"
Tree of Thoughts + RAGExplore multiple reasoning paths, retrieve at each nodeComplex research questions with ambiguous sub-paths
ReAct + retrieval toolLLM issues Thought → Action(search: "query") → Observation → repeatAgent with web search or vector DB as a tool
Corrective RAGAfter generation, evaluate faithfulness; if low, re-retrieve with corrected queryHigh-stakes answer quality; when hallucination is costly
Production Checklist & Common Failure Modes
✅ Production checklist
  • Same embedding model for ingestion and query — enforced at the API layer
  • Chunk overlap of 10–20% so context doesn't break at chunk boundaries
  • Metadata stored alongside each vector (source, timestamp, category)
  • Reranker in the pipeline — bi-encoder retrieval alone is insufficient for precision
  • RAGAS evaluation on a golden dataset before every deployment
  • Faithfulness monitoring in production (sample 5% of responses, run LLM judge)
  • Query caching for identical/near-identical questions (semantic dedup)
❌ Common failure modes
  • Low retrieval recall: chunks too small, embedding model wrong domain — fix: larger chunks + domain-specific embedder
  • Low faithfulness: LLM ignoring context, hallucinating — fix: explicit "answer only from context" system prompt + faithfulness eval
  • Context pollution: irrelevant chunks retrieved — fix: reranker + smaller k
  • Lost in the middle: LLM ignores documents in the middle of a long context — fix: put most relevant chunks first and last
  • Stale knowledge: documents not re-indexed after updates — fix: incremental indexing pipeline on document change events
Common Mistakes & Misconceptions
  • RAG as a silver bullet for hallucination — RAG reduces hallucination but doesn't eliminate it. The LLM can still ignore retrieved context. Faithfulness evaluation is essential
  • Skipping reranking — bi-encoder retrieval has poor precision on hard queries. Cross-encoder reranking is cheap (runs on top-50 docs) and reliably improves quality 5–15%
  • Fixed chunk size for all document types — code, PDFs, tables, prose all have different optimal chunking strategies
  • Not evaluating retrieval separately from generation — if RAGAS scores are low, measure recall@5 on retrieval first. Most RAG failures are retrieval failures, not LLM failures
  • Embedding model mismatch — using a different model to embed queries vs documents. The embedding space must be identical
  • k too large in the prompt — injecting 20 retrieved chunks fills the context window with noise. 3–5 reranked chunks is typically optimal

🎯 Interview Gold

What is RAG? "RAG grounds LLM outputs in retrieved knowledge. At query time: embed the question, run ANN search over indexed document embeddings, retrieve top-k chunks, inject them into the LLM context, generate a grounded answer. The key insight: the LLM's parametric memory is unreliable for facts — retrieval provides real, current, citable information."

Why hybrid search? "Dense retrieval handles semantic similarity but misses exact keywords. BM25 handles keywords but misses paraphrases. Hybrid search fuses both ranked lists via Reciprocal Rank Fusion — consistently outperforms either alone. BGE-M3 generates dense + sparse signals from a single encoder, making the combination simple."

How do you evaluate a RAG pipeline? "I use RAGAS: faithfulness (are claims supported by context?), context recall (did retrieval find the necessary info?), and answer relevance (does the answer address the question?). I build a golden dataset of 50–100 QA pairs from the actual corpus and run these metrics on every pipeline change. Faithfulness is the first metric to check — it's the hallucination measure."

RAG vs fine-tuning? "RAG for knowledge — facts, current information, sourced answers. Fine-tune for behaviour — style, format, persona, task-specific patterns. The combination is the best of both worlds: LoRA shapes how the model communicates, RAG provides what it knows."
