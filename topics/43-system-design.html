ML System Design
Senior-level interview territory — design a fraud detection system, recommendation engine, or credit scoring platform. The framework: requirements → data → features → model → serving → monitoring.
ML System Design Framework
  1. Clarify requirements: latency (real-time vs batch), volume (TPS), accuracy target, update frequency, regulatory constraints
  2. Problem framing: supervised/unsupervised/RL? Binary/multi-class/ranking? Objective metric?
  3. Data strategy: sources, freshness, labels (how collected?), historical volume, class balance
  4. Feature engineering: real-time features (last 5 transactions), pre-computed features (30-day payment rate), embeddings
  5. Model selection: baseline, main approach, why it fits requirements
  6. Training pipeline: offline training schedule, data versioning, experiment tracking
  7. Serving infrastructure: latency budget, scaling, fallback logic
  8. Monitoring & iteration: metrics, drift detection, retraining triggers

Latency vs Accuracy Trade-offs
  • Quantize model (INT8): 2–4× faster inference, <1% accuracy drop
  • Distill large model → small model: smaller, faster, some accuracy cost
  • Feature selection: fewer features → faster inference
  • Approximate neighbors (FAISS) instead of exact: 100× faster, tiny accuracy loss
  • Async enrichment: serve fast with base features, enrich asynchronously for human review queue
Scalability Considerations
  • Stateless serving: each prediction request is independent — easy horizontal scaling
  • Caching: cache predictions for identical inputs (careful — staleness risk)
  • Horizontal scaling: multiple model servers behind a load balancer
  • GPU batching: accumulate requests into batches; GPUs are efficient at batch operations
  • Database bottleneck: feature lookups often limit throughput — use Redis for feature serving
🎯 Interview Gold

"When I get a system design problem, I start by anchoring on two things: (1) what latency is acceptable — real-time (<100ms) forces very different choices than batch (hours), and (2) what does failure look like — a wrong fraud decision means customer friction, a wrong credit decision means money loss. Then I work through the data flywheel: more decisions → more labeled data → better model → better decisions. The biggest scaling trap is model serving becoming a monolith — separate the model server from the feature server from the rule engine."
