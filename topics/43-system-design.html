ML System Design
Senior-level interview territory — design a fraud detection system, recommendation engine, real-time search ranking platform, or LLM inference pipeline. The framework: requirements → data → features → model → serving → monitoring. This topic covers three complete case studies.
ML System Design Framework (Apply to Every Problem)
  1. Clarify requirements: latency (real-time vs batch), volume (TPS), accuracy target, update frequency, regulatory constraints
  2. Problem framing: supervised / unsupervised / RL? Binary / multi-class / ranking? What is the ground truth / label source?
  3. Data strategy: sources, freshness, labels (how collected?), class balance, historical volume
  4. Feature engineering: real-time features (last 5 transactions), pre-computed features (30-day rate), embeddings
  5. Model selection: justify the architecture — why does it fit the constraints?
  6. Training pipeline: offline training schedule, data versioning, experiment tracking
  7. Serving infrastructure: latency budget per layer, scaling strategy, fallback logic
  8. Monitoring & iteration: metrics, drift detection, retraining triggers, A/B testing

Class Imbalance Strategy
  • Fraud rate is typically 0.1–1% — severe class imbalance
  • Don't oversample at training time — adjust decision threshold instead
  • Use scale_pos_weight in LightGBM (= negatives / positives)
  • Evaluate with PR-AUC not ROC-AUC — ROC is misleading under heavy imbalance
  • Business metric: cost matrix (false negative = fraud loss; false positive = customer friction)
Case Study 2 — Search & Ranking System
Two-Stage Retrieval + Ranking Architecture
All large-scale search and recommendation systems use a two-stage funnel: cheap retrieval (millions → hundreds of candidates), then expensive ranking (hundreds → ranked top-k).
Search pipeline: query → retrieval (ANN + BM25) → rerank (pointwise LTR) → serve
Retrieval Layer Details
  • Dense retrieval (ANN): embed query + docs with same bi-encoder (e.g., E5, BGE); find nearest neighbours with FAISS/HNSW; handles semantic similarity
  • Sparse retrieval (BM25): TF-IDF-based keyword matching; fast via inverted index; misses synonyms
  • Hybrid search: combine dense + sparse scores (RRF or learned weighting); outperforms either alone on most benchmarks
  • HNSW index: hierarchical navigable small world graph — sub-linear ANN search, build once, query in ~1ms
Ranking (LTR) Details
  • Learning-to-rank (LTR): given (query, doc) pairs, predict a relevance score
  • Pointwise: binary relevant / not (simplest; ignores relative order)
  • Pairwise (RankNet, LambdaMART): model prefers doc A over doc B — directly optimises ranking
  • Features: BM25 score, semantic similarity, CTR, freshness, user affinity, PageRank, price
  • Label source: human relevance judgments + click-through data (clicks = implicit feedback, noisy but scalable)
  • Offline metric: NDCG@10; online metric: click-through rate, session engagement

Two-Tower Architecture
$$\begin{aligned} e_u &= f_\theta(u_\text{features}) \quad \text{[user tower — trained once per user per hour]} \\ e_i &= g_\phi(i_\text{features}) \quad \text{[item tower — pre-computed offline]} \\ \hat{y} &= e_u \cdot e_i \quad \text{[dot product similarity → ANN retrieval]} \end{aligned}$$
  • Train with in-batch negatives: other items in the same batch serve as negatives — no explicit negative sampling needed
  • Hard negatives (high-scoring but not interacted) significantly improve retrieval quality
Data Flywheel
The core competitive advantage in recommendation systems is the feedback loop:
  • Better recommendations → more engagement → more interaction data
  • More interaction data → better model → better recommendations
  • This compounds over time — the data flywheel is what makes Netflix/Spotify hard to replicate
  • Position bias: items shown higher get more clicks regardless of quality — must correct for this in training (inverse propensity weighting)
Cross-Cutting: Latency vs Accuracy Trade-offs & Scalability
Latency vs Accuracy
  • Quantise model (INT8): 2–4× faster inference, <1% accuracy drop
  • Distill large model: smaller, faster, some accuracy cost
  • Approximate neighbours (FAISS): 100× faster than exact; tiny accuracy loss
  • Async enrichment: serve fast with base features; enrich asynchronously for human review queue
  • Feature precomputation: move expensive features offline to feature store
Scalability Patterns
  • Stateless serving: each prediction independent → horizontal scaling trivial
  • Caching: cache predictions for identical inputs — careful of staleness
  • GPU batching: accumulate requests; GPUs are highly efficient at batch operations
  • Redis for feature serving: database lookups often limit throughput more than model
  • Read replicas: scale read-heavy feature serving independently of write path
Common Mistakes & Misconceptions
  • Jumping to model architecture without clarifying requirements — the correct model for fraud detection at <50ms real-time is completely different from a nightly batch fraud scoring job. Always start with: latency requirements, throughput, label availability, acceptable false positive rate, and failure cost. These constraints determine everything
  • Single-stage retrieval for large-scale ranking — scoring all items with a complex model at serving time is O(n) × model cost. Two-stage is the production standard: cheap approximate retrieval (ANN, heuristic rules) to get top-K candidates, then expensive reranking on those K items only
  • Letting the feature store become the latency bottleneck — high-latency feature lookups (joining user history from a data warehouse at serving time) blow up P99 latency. High-cardinality features (user embedding, session stats) must be precomputed and served from Redis/Memcached at <5ms, not computed on-demand
  • Training on biased implicit feedback labels — clicks and purchases reflect both user preference AND item position/visibility (position bias). A model trained naively on clicks learns to favour already-prominent items in a feedback loop. Always debias labels: inverse propensity weighting for position bias; randomisation experiments to collect unbiased signal
  • Ignoring cold-start in the design — new users have no history; new items have no engagement data. The system must degrade gracefully: rule-based fallbacks, content-based features (item metadata), or population-level priors. Cold-start is always asked in system design interviews; have an explicit strategy
How It's Used at Top Companies
  • Netflix (recommendation): two-tower retrieval (user embedding × item embedding → ANN) produces top-500 candidates; reranked by a 200-feature XGBoost model; final diversity post-processing (no two items from same series adjacent) — 3-stage pipeline serving 200M+ users
  • Twitter/X (feed ranking): candidate sources (follows, trends, ads, recommendations) → heavy ranker (neural net on 1000+ features) → heuristic filters (content policy) → mixer (diversity enforcement) — 4-stage pipeline with each stage progressively more expensive
  • DoorDash (ETA prediction): real-time ML system with restaurant prep time model + delivery route model + driver supply model combined into ETA; features served from Redis (restaurant historical prep times), computed at order time in <200ms
  • Stripe (fraud at auth time): XGBoost scoring in <50ms at payment authorization; features from Redis feature store (velocity features precomputed on rolling windows), graph features (merchant-card-IP network), and request features computed inline — hard latency SLA enforced at API gateway
🎯 Interview Gold

What to say: "For any ML system design I start with two questions: (1) what latency is required — real-time (<100ms) forces pre-computed features, lightweight models, and ANN retrieval; batch (hours) allows expensive feature computation and complex models; (2) what does failure look like — a wrong fraud block costs customer trust; a missed fraud costs money. For ranking I default to two-stage: cheap ANN retrieval → expensive reranking. The biggest scaling trap: feature store DB lookups at serving time — move high-cardinality features offline into Redis."

Common mistake to avoid saying: "I'd train a neural network on user clicks." — Clicks have position bias (users click what's shown prominently regardless of preference). You need inverse propensity weighting or randomisation experiments to get unbiased labels. Naive click training creates a feedback loop.

Expect the follow-up: "How would you handle position bias in a recommendation system?" — Answer: (1) Inverse propensity weighting — weight each training example by 1/P(item shown at that position); items shown prominently get down-weighted. (2) Position-aware features — include the impression position as a feature at training time, then set it to a neutral value (e.g., position 1) at serving time; model learns to remove the bias. (3) Randomisation — occasionally inject random items at random positions; these clicks are unbiased ground truth for training.
