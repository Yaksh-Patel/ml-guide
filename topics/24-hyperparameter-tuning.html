Hyperparameter Tuning
The systematic search for the model configuration that maximizes validation performance — done right, it can be the difference between a good model and a great one.
Grid Search
Exhaustively evaluate all combinations of a predefined hyperparameter grid.
  • Simple, reproducible, parallelizable
  • Weakness: exponential in number of hyperparameters — 5 params × 10 values = 100,000 evaluations
  • Use when: small grid, fast model, few hyperparameters
Random Search
Sample configurations randomly from distributions over each hyperparameter.
  • Covers more diverse configurations in same budget
  • Key insight: if only a few hyperparameters matter, random search wastes less budget on the unimportant ones
  • Typically 10x more efficient than grid search
  • Bergstra & Bengio (2012): random > grid for most practical cases
Bayesian Optimization
Build a probabilistic model (surrogate) of the objective function. Use it to select next configurations that maximize expected improvement.
$$\begin{aligned} \text{Acquisition (EI):} & \quad EI(x) = \mathbb{E}[\max(f(x)-f^*,\, 0)] \\ \text{Surrogate:} & \quad \text{Gaussian Process (GP) or TPE (Tree Parzen Estimator)} \end{aligned}$$
  • Samples more from promising regions, less from bad ones
  • Each evaluation informs the next — learns the landscape
  • Best for expensive models (each trial takes minutes/hours)
  • Tools: Optuna (TPE), Hyperopt, Ray Tune, W&B Sweeps
Optuna example: study = optuna.create_study(); study.optimize(objective, n_trials=100)
Hyperopt: One of the earliest Bayesian optimisation libraries. Uses TPE by default and integrates with MLflow. fmin(objective, space, algo=tpe.suggest, max_evals=100). Optuna is generally preferred today for its cleaner API and pruning support, but Hyperopt remains widely used in legacy codebases.
Best Practices
  • Search order: learning_rate and n_estimators first (most impactful), then regularization params, then minor ones
  • Early stopping in each trial: don't waste budget on clearly bad configs
  • Log scale for learning rate: sample from [1e-4, 1e-1] in log space, not linearly
  • Nested CV for unbiased evaluation: inner loop for tuning, outer loop for evaluation
  • Budget allocation: 50% Bayesian, 50% manual exploration around best found region
Common Mistakes & Misconceptions
  • Grid searching on a linear scale — learning rate of 0.001 vs 0.01 is a 10× difference; 0.1 vs 0.2 is only 2×. Always use log-scale sampling for any hyperparameter spanning orders of magnitude: learning_rate, λ, C, gamma
  • Using the test set for hyperparameter selection — tuning hyperparameters on the test set inflates reported performance and invalidates the generalisation estimate. Always tune on a held-out validation set or via CV; test set is touched exactly once
  • Running too few Bayesian optimization trials — Bayesian optimization needs ~20 "warm-up" trials before the surrogate model is reliable. Running only 10 total trials gives results no better than random. Minimum 50–100 trials for meaningful Bayesian optimization
  • Not setting early stopping inside each trial — each trial of a gradient boosting model should use early stopping so bad configs terminate early. Without this, every trial wastes the full n_estimators budget regardless of whether the config is promising
  • Ignoring interaction effects between hyperparameters — learning_rate and n_estimators are strongly coupled (lower LR needs more trees). Grid search over them independently misses the interaction. Bayesian optimization handles this naturally
How It's Used at Top Companies
  • Google (Vizier / Vertex AI Hyperparameter Tuning): Bayesian optimization at scale — internally used to tune all major Google ML models; exposed as Vertex AI Hyperparameter Tuning with Gaussian Process or NSGA-II (multi-objective) backends
  • Meta (Ax / BoTorch): open-sourced their Bayesian optimization platform (Ax + BoTorch); used internally for model tuning and A/B test design; state-of-the-art Gaussian process surrogates with GPU acceleration
  • Kaggle top solutions: Optuna with TPE sampler is the de facto standard for competition hyperparameter tuning; pruning (MedianPruner) kills bad trials early, dramatically increasing search efficiency
  • MLOps pipelines (Airflow + MLflow): hyperparameter tuning integrated into CI/CD — each model retraining run triggers a tuning stage that searches within pre-defined bounds, logs results to MLflow, and promotes the best config
🎯 Interview Gold

What to say: "My tuning strategy: (1) fix n_estimators large + early stopping — tree count is handled automatically. (2) Optuna with TPE sampler for Bayesian optimization — more efficient than grid search, handles parameter interactions. (3) Start broad (wide ranges, 50 trials), then refine around the best region (narrow ranges, 50 more trials). (4) Always log-scale for learning_rate, λ, gamma. For XGBoost, learning_rate + max_depth give the most signal per trial. For neural nets, learning_rate is #1, then batch size, then architecture."

Common mistake to avoid saying: "I grid searched over [0.1, 0.01, 0.001] for learning rate." — This is 3 values with 10× spacing on log scale. Bayesian optimization explores continuously and adapts to promising regions. Grid search on 3 values is barely better than guessing.

Expect the follow-up: "How does Bayesian optimization work?" — Answer: maintains a surrogate model (usually Gaussian Process) of the objective function over the hyperparameter space. After each trial, updates the surrogate with the new result. Uses an acquisition function (Expected Improvement, UCB) to select the next point that balances exploration (uncertain regions) vs exploitation (near the current best). Each trial is informed by all previous results — far more efficient than random or grid search.
