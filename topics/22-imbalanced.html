Handling Imbalanced Datasets
In fraud detection (0.1% positive), cancer detection (5%), and credit default (3%), naive models predict "no" for everything and get 99% accuracy. The metric and data preparation must account for the imbalance.
Resampling Strategies
  • Random oversampling: duplicate minority class samples. Simple but overfits on exact duplicates.
  • SMOTE: Synthetic Minority Oversampling TEchnique — interpolate between minority class neighbors to create new synthetic points. Better than random oversampling.
  • ADASYN: Adaptive Synthetic — generates more samples where misclassification is likely.
  • Random undersampling: remove majority class samples. Loses information — only use when data is huge.
  • Tomek links: remove boundary majority samples (cleaning, not balancing).
Algorithm-Level Adjustments
  • Class weights: weight=class_weight='balanced' in sklearn — scales the loss contribution. No data loss. My #1 first try.
  • Threshold tuning: instead of default 0.5 cutoff, find threshold that maximizes F1 or business metric on validation set.
  • Cost-sensitive learning: assign asymmetric misclassification costs — missed positive costs more than false alarm.
  • Focal Loss: down-weights easy negative examples dynamically. Used in RetinaNet (object detection) and imbalanced NLP tasks.

📖 Fraud Analogy

If 0.1% of transactions are fraud and you predict "not fraud" for everything: 99.9% accuracy. 0% recall. 100% of fraudsters escape. Accuracy is a lie. Your PR-AUC baseline is 0.001. Your model must beat that to add any value.

Common Mistakes & Misconceptions
  • Applying SMOTE to the whole dataset before splitting — synthetic samples from the test set contaminate training. Always split first, then apply SMOTE only to the training fold. The test set must contain only real examples
  • "Random oversampling fixes imbalance" — it duplicates exact examples, causing the model to overfit to those specific training points. SMOTE creates synthetic interpolations between existing minority examples — always prefer SMOTE over random oversampling
  • Not tuning the decision threshold after training — the default 0.5 threshold is almost never optimal for imbalanced problems. After training, find the threshold that maximises your business metric (e.g., F1, or recall@fixed precision) on the validation set
  • Undersampling as the first resort — discarding majority examples loses real information. class_weight='balanced' achieves equivalent loss re-weighting without data loss — try that first
  • Ignoring the imbalance in CV folds — always use StratifiedKFold; a random fold with 0 positive examples will produce nonsensical metrics and unstable gradient estimates during training
How It's Used at Top Companies
  • Stripe / PayPal (fraud detection): class_weight or scale_pos_weight as the first adjustment; SMOTE-ENN (SMOTE + edited nearest neighbours cleaning) for borderline cases; PR-AUC as primary metric; threshold tuned to match regulatory recall requirements
  • Google (spam detection): Focal Loss in neural models — dynamically down-weights easy negatives so training focuses on hard positive examples; standard in imbalanced NLP classification
  • Healthcare AI (rare disease detection): extreme imbalance (1:10000); cost-sensitive learning with asymmetric loss weights; human review triggered above a low-recall threshold to avoid missing rare cases
  • Insurance (claim fraud): XGBoost with scale_pos_weight + threshold optimised for F-beta (β=2, weighting recall 2× over precision) — missing fraud is more costly than false accusations
🎯 Interview Gold

What to say: "My imbalanced data strategy in order: (1) PR-AUC as primary metric — not accuracy, not AUC-ROC alone. (2) class_weight='balanced' first — no data manipulation, same information. (3) If insufficient, SMOTE on training set only — never on validation/test. (4) Tune decision threshold on validation set to maximise the business metric (e.g., recall at acceptable precision). (5) Focal Loss for neural nets — focuses training on hard examples. Always StratifiedKFold to maintain class ratio in every CV fold."

Common mistake to avoid saying: "I applied SMOTE before train/test split." — This is data leakage. Synthetic minority examples generated from test data contaminate training. SMOTE must happen inside the CV loop, after splitting.

Expect the follow-up: "How does SMOTE work?" — Answer: for each minority example, find its k nearest minority-class neighbours. Pick one at random, then generate a new synthetic example by interpolating along the line between the original point and its neighbour: x_new = x + λ·(x_neighbour − x) where λ ~ Uniform(0,1). Creates new examples in the minority manifold rather than duplicating existing ones.
