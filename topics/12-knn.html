k-Nearest Neighbors (KNN)
The laziest learner in ML — literally stores all training data and defers all computation to prediction time. No training phase, no parameters.
How It Works
  1. Store all training examples (X_train, y_train)
  2. At prediction: find the k nearest neighbors of query point x using a distance metric
  3. Vote: majority class (classification) or mean of k labels (regression)
$$\begin{aligned} d_{\text{euclidean}}(\mathbf{a},\mathbf{b}) &= \sqrt{\textstyle\sum_i (a_i - b_i)^2} \\ d_{\text{manhattan}}(\mathbf{a},\mathbf{b}) &= \textstyle\sum_i |a_i - b_i| \\ d_{\text{cosine}}(\mathbf{a},\mathbf{b}) &= 1 - \dfrac{\mathbf{a}\cdot\mathbf{b}}{|\mathbf{a}||\mathbf{b}|} \end{aligned}$$
Strengths
  • Zero training time
  • Non-parametric — no assumptions about data distribution
  • Naturally multi-class
  • Simple to understand and implement
  • Works well for low-dimensional, small datasets
Weaknesses
  • Curse of dimensionality: in high-D, everything is equidistant
  • Slow prediction: O(n) per query (use KD-tree or Ball-tree for speed)
  • Memory-intensive: stores entire training set
  • Sensitive to irrelevant features — feature scaling required
  • No feature importance output
Choosing k
  • Small k (e.g., k=1): low bias, high variance — overfits, noisy boundaries
  • Large k: high bias, low variance — over-smooths boundaries
  • Rule of thumb: k = √n as starting point; tune via CV
  • Use odd k for binary classification to avoid ties
📖 New Neighborhood Analogy

You move to a new city and want to know if a neighborhood is safe. KNN asks: "What are the 5 closest data points (past reports) to this location? If 4/5 say unsafe, predict unsafe." It's pure pattern matching — no model, just memory. Works great if you have good data; fails if data is sparse or high-dimensional.

Common Mistakes & Misconceptions
  • Forgetting to scale features — the #1 KNN mistake. A feature ranging [0,1000] completely drowns one ranging [0,1] in Euclidean distance. Always StandardScaler or MinMaxScaler before KNN
  • Using k=1 in noisy data — 1-NN is extremely sensitive to outliers and label noise; it perfectly memorises training data (zero training error, poor generalisation). Always tune k via CV
  • Brute-force KNN at scale — O(n·d) per query. For n > 50k, use KD-tree (works for d < 20) or Ball-tree; for high-dimensional data or n > 1M use approximate nearest neighbors (Faiss, HNSW)
  • "Curse of dimensionality" is just a phrase — in practice: when d > 20, Euclidean distance concentrates around the same value for all points (ratio of max to min distance → 1). KNN becomes random beyond ~30 dimensions without dimensionality reduction first
  • Using KNN for large-scale production serving — storing the entire training set (no compression) and O(n) prediction time make raw KNN unviable at scale. Use ANN indexes (Faiss, HNSW, ScaNN) for production
How It's Used at Top Companies
  • Spotify / Netflix (recommendation): not raw KNN but approximate KNN (HNSW via Faiss) over learned item/user embeddings — "nearest neighbors in embedding space" is the core retrieval step in modern recommender systems
  • Google (image search): ScaNN (Google's ANN library) performs billion-scale nearest neighbor search over image embeddings for reverse image search and visual similarity
  • Fraud detection (anomaly): 1-NN distance as anomaly score — a point with no close neighbors in training space is a novel/anomalous transaction
  • Drug discovery: molecular similarity search uses KNN over Morgan fingerprints (chemical feature vectors) to find similar known compounds to a candidate molecule
🎯 Interview Gold

What to say: "KNN is a powerful concept but not a production algorithm in raw form — O(n) prediction time and full training set storage are non-starters at scale. Its real importance today is as the theoretical basis for approximate nearest neighbor search: embedding everything into a dense vector space and retrieving similar items via ANN (Faiss, HNSW). That pattern powers recommendation retrieval, semantic search, and RAG. For KNN to work at all: always scale features, tune k via CV, and use odd k for binary classification to avoid ties."

Common mistake to avoid saying: "KNN is slow." — The right answer is more nuanced: brute-force KNN is slow, but ANN indexes (HNSW, IVF) achieve sub-millisecond query time even on billions of vectors. KNN's concept is extremely modern and in production everywhere.

Expect the follow-up: "How would you scale KNN to 100 million items?" — Answer: build an HNSW index (e.g., via Faiss or hnswlib); it gives approximate but extremely fast results (sub-millisecond at scale). Trade recall for speed — HNSW at ef=200 gets 99%+ recall at ~1ms latency.
