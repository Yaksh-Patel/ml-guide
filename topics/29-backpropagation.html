Backpropagation & Training Loop
Efficient computation of all gradients using the chain rule, plus the complete gradient descent training loop — from raw data to updated weights. Once you can trace this by hand, backprop becomes mechanical rather than magical.
The Complete Training Loop
🐍 Python
for epoch in range(num_epochs):
    for X_batch, y_batch in dataloader:        # Step 1: mini-batch
        # --- FORWARD PASS ---
        z1 = X_batch @ W1 + b1                 # Step 2a: linear transform
        h1 = relu(z1)                           # Step 2b: non-linearity
        z2 = h1 @ W2 + b2                       # Step 2c: output layer
        y_hat = sigmoid(z2)                     # Step 2d: probability

        # --- LOSS ---
        loss = binary_cross_entropy(y_hat, y_batch)   # Step 3: scalar loss

        # --- BACKWARD PASS ---
        optimizer.zero_grad()                   # Step 4a: clear old gradients!
        loss.backward()                         # Step 4b: compute all ∂L/∂W

        # --- CLIP (optional but critical for RNNs/Transformers) ---
        clip_grad_norm_(model.parameters(), max_norm=1.0)  # Step 5

        # --- UPDATE ---
        optimizer.step()                        # Step 6: W ← W − α∇L

    # --- VALIDATION ---
    model.eval()
    with torch.no_grad():                       # Step 7: no grad tracking
        val_loss = evaluate(model, val_loader)
    model.train()

Critical Steps Often Missed
  • zero_grad(): PyTorch accumulates gradients — missing this adds previous batch's gradients to the current one. Silent, hard-to-detect bug.
  • model.eval() for validation: disables Dropout and uses BatchNorm running statistics. Missing this corrupts validation metrics.
  • torch.no_grad(): stops building the computation graph at inference — 2–3× memory + speed improvement.
  • Gradient clipping: essential for RNNs and Transformers — caps gradient norm to prevent a bad batch from blowing up weights.
Chain Rule on a Computational Graph
$$\begin{aligned} \text{For } L = \text{softmax}(W^2 \cdot \text{ReLU}(W^1 x + b^1) + b^2): \\ \frac{\partial L}{\partial W^2} = \frac{\partial L}{\partial \hat{y}} \cdot \frac{\partial \hat{y}}{\partial z^2} \quad \text{(output layer gradient)} \\ \frac{\partial L}{\partial W^1} = \frac{\partial L}{\partial \hat{y}} \cdot \frac{\partial \hat{y}}{\partial z^2} \cdot \frac{\partial z^2}{\partial h^1} \cdot \frac{\partial h^1}{\partial z^1} \cdot \frac{\partial z^1}{\partial W^1} \\ \frac{\partial h^1}{\partial z^1} = \mathbb{1}(z^1>0) \quad \text{(ReLU local gradient)} \\ \frac{\partial z^1}{\partial W^1} = x \quad \text{(linear local gradient)} \end{aligned}$$
Key insight: during the forward pass, cache the intermediate values. During the backward pass, multiply them in reverse order. This makes backprop O(2 × forward pass), not O(parameters).
Backprop: gradients flow backward layer by layer via chain rule
Concrete 2-Layer Gradient Computation (Step-by-Step)
$$\begin{aligned} \text{Architecture: } x\,(2D) \rightarrow h\,(3D,\text{ReLU}) \rightarrow \hat{y}\,(1D,\text{sigmoid}) \\ \textbf{--- OUTPUT LAYER ---} \\ \frac{\partial L}{\partial \hat{y}} = \frac{\hat{y}-y}{\hat{y}(1-\hat{y})} \quad \text{(BCE w.r.t. } \hat{y}\text{)} \\ \frac{\partial \hat{y}}{\partial z^2} = \hat{y}(1-\hat{y}) \quad \text{(sigmoid derivative)} \\ \delta^2 = \frac{\partial L}{\partial z^2} = \hat{y}-y \quad \text{[these cancel nicely!]} \\ \frac{\partial L}{\partial W^2} = \delta^2 \cdot (h^1)^T \quad \text{(outer product)} \\ \frac{\partial L}{\partial b^2} = \delta^2 \\ \textbf{--- HIDDEN LAYER ---} \\ \frac{\partial L}{\partial h^1} = (W^2)^T \cdot \delta^2 \quad \text{[}W^{2T}\text{ routes gradient back]} \\ \frac{\partial h^1}{\partial z^1} = \mathbb{1}(z^1>0) \quad \text{(ReLU: 1 if positive, 0 if not)} \\ \delta^1 = \frac{\partial L}{\partial z^1} = (W^2)^T\cdot\delta^2 \odot \mathbb{1}(z^1>0) \\ \frac{\partial L}{\partial W^1} = \delta^1 \cdot x^T \\ \frac{\partial L}{\partial b^1} = \delta^1 \\ \textbf{--- WEIGHT UPDATE ---} \\ W^2 \leftarrow W^2 - \alpha \cdot \frac{\partial L}{\partial W^2} \\ W^1 \leftarrow W^1 - \alpha \cdot \frac{\partial L}{\partial W^1} \end{aligned}$$
Vanishing Gradients
  • Gradients → 0 as they multiply through many saturating activations (sigmoid, tanh)
  • Early layers learn nothing — training stalls
  • Fixes: ReLU activations, BatchNorm, residual connections (ResNets), careful init
Exploding Gradients
  • Gradients grow exponentially → weights become NaN
  • Common in RNNs with long sequences
  • Fixes: gradient clipping (clip_grad_norm), lower learning rate, BatchNorm, weight decay
BatchNorm — The Stability Breakthrough
$$\begin{aligned} \text{BatchNorm}(x) = \gamma \cdot \frac{x - \mu_B}{\sqrt{\sigma^2_B + \varepsilon}} + \beta \\ \mu_B = \text{batch mean},\quad \sigma^2_B = \text{batch variance} \\ \gamma,\,\beta = \text{learned scale and shift parameters} \end{aligned}$$
  • Normalizes activations per mini-batch — keeps gradients in a healthy range
  • Acts as regularizer (noisy normalization during training)
  • Allows higher learning rates — training is much faster and stable
  • Applied before or after activation (debate; pre-activation ResNets say before)
What Autograd Does For You
In PyTorch, loss.backward() executes the entire chain automatically by traversing the computation graph in reverse. Each operation records its local gradient during the forward pass, then multiplies them in chain-rule order during backward.
🐍 Python
x = torch.tensor([...], requires_grad=True)
z1 = x @ W1 + b1          # records: ∂z1/∂W1 = x
h  = z1.clamp(min=0)       # records: ∂h/∂z1 = 1(z1>0)
z2 = h @ W2 + b2
loss = F.binary_cross_entropy(z2.sigmoid(), y)
loss.backward()            # traverses graph, accumulates grads
print(W1.grad)             # ∂L/∂W1 is now populated
Computational Graph
  • Every tensor operation creates a node in the graph
  • Each node stores: its output value + its local gradient function
  • backward() walks this DAG in reverse topological order
  • detach(): cuts gradient flow (e.g., for target network in RL)
  • retain_graph=True: allows multiple backward passes (used in GANs)
  • torch.no_grad(): don't build graph at all (inference mode)

📖 Telephone Game

Backprop is feedback traveling backward through 100 student pairs whispering to each other. By the time it reaches student #1 (first layer), the message is either completely garbled (vanishing) or shouted so loud it's noise (exploding). BatchNorm is a relay station every 10 pairs ensuring the message arrives intact and at the right volume.

Common Mistakes & Misconceptions
  • "Backprop is O(parameters)" — wrong. Backprop is O(2 × forward pass) due to the chain rule reusing intermediate activations. The cost of a backward pass is roughly 2× a forward pass regardless of depth, because each layer's gradient reuses the already-computed activations stored during the forward pass
  • Vanishing gradients only happen in RNNs — they happen in any deep network with saturating activations (sigmoid, tanh) in hidden layers. The fix is the same: ReLU/GELU activations + BatchNorm + residual connections. RNNs are just the most severe case
  • Exploding gradients only affect training stability visually — NaN loss or wildly oscillating training curves are the obvious sign, but subtle exploding gradients can silently prevent convergence. Always monitor gradient norms; clip at a threshold (e.g., max_norm=1.0 in PyTorch)
  • Not zeroing gradients between batches in PyTorch — PyTorch accumulates gradients by default. Always call optimizer.zero_grad() before loss.backward(). Forgetting this causes gradients to accumulate across batches, effectively using a much larger batch size than intended
  • Confusing backprop with gradient descent — backprop computes the gradients (the what). Gradient descent uses those gradients to update weights (the how). They are separate: backprop is an efficient algorithm for chain rule application; gradient descent is the optimisation strategy
How It's Used at Top Companies
  • PyTorch autograd (Meta): backprop is implemented via dynamic computational graphs — every operation records a backward function; calling loss.backward() traverses this graph in reverse. This "define-by-run" approach enables variable-length inputs (NLP) and conditional computation
  • Google (TPU training): gradient checkpointing used at scale — instead of storing all activations for backprop, recompute them during the backward pass, trading 33% compute overhead for 10× memory savings on billion-parameter models
  • Transformer training (all major labs): gradient clipping (clip_grad_norm=1.0) is universal in transformer training recipes — prevents occasional large gradient spikes from destabilising attention layer training
  • Mixed-precision training (NVIDIA): forward pass and activations stored in FP16 (half memory, faster); gradients computed in FP16 then scaled to FP32 for the weight update — gradient scaling prevents FP16 underflow in the backward pass
🎯 Interview Gold

What to say: "Backprop is O(2 × forward pass) — not O(parameters) — because of shared computation via the chain rule. The mathematical elegance: ∂L/∂z = ŷ − y for BCE+sigmoid — the sigmoid derivative cancels perfectly with the BCE derivative, leaving a clean residual gradient. Residual connections (ResNet) create a gradient highway — the skip connection passes gradients directly to early layers without multiplicative attenuation, solving the vanishing gradient problem for depth. This is why ResNet-152 trained when a plain 152-layer net couldn't."

Common mistake to avoid saying: "I forgot to zero the gradients." — In PyTorch, optimizer.zero_grad() before every backward pass is fundamental. Mentioning this in an interview without being asked signals attention to implementation details.

Expect the follow-up: "How does gradient checkpointing work?" — Answer: normal backprop stores all layer activations during the forward pass to use in the backward pass (memory = O(depth)). Gradient checkpointing only stores activations at "checkpoint" layers; intermediate activations are recomputed during the backward pass when needed. Trades ~33% extra compute for O(√depth) memory — essential for training very deep models or very long sequences on limited GPU memory.
