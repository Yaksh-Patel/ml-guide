Cross-Validation
The honest way to estimate how your model will perform on unseen data — without wasting data on a fixed test set during development.
k-Fold Cross-Validation
Split data into k equal folds. Train on k−1 folds, validate on the held-out fold. Rotate k times. Average the k validation scores.
5-Fold Cross-Validation — each fold takes a turn as validation

Common CV Mistakes
  • Data leakage from preprocessing: fit scaler on all data before splitting → test info leaks into training
  • Peeking at test set: using test performance to make modeling decisions inflates estimates
  • No stratification: imbalanced classes can produce folds with 0 positives
  • Time series with random split: future data leaks into training — always use temporal split
⚠️ The Golden Rule

Preprocessing (scaling, imputation, encoding) must be fit ONLY on the training fold, then applied to validation. Use sklearn Pipelines to enforce this automatically. Never fit on the full dataset before splitting.

Common Mistakes & Misconceptions
  • Fitting preprocessors on full data before CV — the #1 leakage source. StandardScaler, imputers, encoders must all be inside a Pipeline; fit only on train fold, transform both folds. Even seemingly harmless steps (e.g., removing constant features) can leak if done globally
  • "I used k=10 for more precision" — higher k increases computation k-fold and adds variance to the CV estimate (each fold has less test data). k=5 is the empirical sweet spot between bias and variance of the CV estimate itself
  • Using random k-fold on imbalanced data — a fold might have 0 positive examples. Always use StratifiedKFold for classification; it maintains class proportions in each fold
  • Tuning hyperparameters on the test set — the test set must be touched exactly once, after all modeling decisions are made. Using it for threshold selection, model choice, or feature selection invalidates the estimate. Use a validation set or nested CV
  • Ignoring temporal leakage in time series — random CV on time series lets future data train the model. Use TimeSeriesSplit (expanding window) or a strict time-based hold-out. Even "random" feature creation can leak future information
How It's Used at Top Companies
  • Kaggle competitions: 5-fold stratified CV is the universal standard; competitors build ensemble stacks using out-of-fold predictions (OOF) from level-1 models as features for level-2 meta-learners — requires rigorous CV discipline to avoid leakage
  • Meta / Google (A/B testing): CV logic applied to time-series experiment evaluation — holdout periods ("holdback windows") used instead of random folds to prevent temporal leakage in metric estimation
  • Stripe (production ML): models re-validated monthly with walk-forward CV (train on months 1–N, validate on month N+1, repeat) to detect concept drift before promoting to production
  • Healthcare (FDA submission): nested CV required by FDA guidance for biomarker validation — outer loop estimates generalisation, inner loop tunes hyperparameters; prevents optimistic bias in reported performance
🎯 Interview Gold

What to say: "For classification I use stratified 5-fold CV — maintains class proportions per fold and gives a reliable generalisation estimate. All preprocessing (scaling, imputation, encoding) lives inside a sklearn Pipeline, which is refitted on each training fold automatically. For time series, I use TimeSeriesSplit with an expanding window — no future leakage. For hyperparameter tuning + generalisation estimate simultaneously, nested CV: inner loop tunes params, outer loop estimates test performance. Expensive but unbiased."

Common mistake to avoid saying: "I scaled all my features, then ran cross-validation." — This is data leakage. Validation fold statistics bled into the scaler. The correct answer is always: Pipeline wraps everything.

Expect the follow-up: "What's the difference between k-fold CV and a simple train/validation/test split?" — Answer: k-fold uses all data for both training and validation (across k iterations), giving a lower-variance generalisation estimate. A single validation split wastes data and its estimate has high variance. Test set is separate in both cases — it's the final unbiased evaluation, not a validation tool.
