Feature Engineering & Data Processing
The art and science of turning raw data into signals that algorithms can use — arguably the highest-ROI activity in applied ML. Better features beat better algorithms every time.

⚠️ Types of Missingness

MCAR (completely at random): safe to drop. MAR (at random given other vars): impute. MNAR (not at random — the value is missing because of what it is): add missingness indicator as feature!

Feature Scaling
Standardization (Z-score)
$$x' = (x - \mu) / \sigma \rightarrow N(0,1)$$
Use for: linear models, SVM, PCA, neural networks. Does NOT bound to [0,1] — outliers still present.
Min-Max Normalization
$$x' = (x - xmin) / (xmax - xmin) \rightarrow \quad \text{[0,1]}$$
Use for: neural networks when you need bounded input. Sensitive to outliers — one extreme value squashes everything else.
Never scale: tree-based models (XGBoost, RF, decision trees) — splits are rank-based, scale doesn't matter. Always scale: KNN, SVM, linear/logistic, neural networks, PCA.

Common Mistakes & Misconceptions
  • Target encoding leakage — the #1 feature engineering mistake. If you encode a category with the mean of the target on the full dataset, target info leaks into features. Always use CV-in-fold target encoding (compute mean from training fold only for each validation fold) or add Laplace/Gaussian noise
  • Creating features from the test set — any feature that uses test set statistics (mean, max, percentile) is leakage. Compute all aggregations from training data, then join to test. Even seemingly innocent group-level aggregations need care
  • Not scaling numeric features before interaction terms — interaction x₁·x₂ is dominated by the larger-scale feature if they're not normalized first. Scale before creating polynomial/interaction features
  • One-hot encoding high-cardinality features — a city with 10,000 unique values creates 10,000 binary columns. Use target encoding (with leakage protection), frequency encoding, or embeddings instead
  • Log-transforming without checking zeros — log(0) = -∞. Always use log(1 + x) or check for zeros before applying log transforms to count data or skewed monetary features
How It's Used at Top Companies
  • Uber (demand forecasting): hour-of-day × day-of-week interaction features, lag features (demand 1 week ago same hour), Fourier series features for weekly/yearly seasonality — hand-crafted temporal features remain critical even in deep learning pipelines
  • Airbnb (search ranking): listing quality score = weighted combination of review count, response rate, booking conversion rate — a single engineered feature that outperforms all raw inputs in ranking models
  • Stripe (fraud detection): velocity features — transactions_last_1h, transactions_last_24h, distinct_merchants_last_7d — rolling aggregations computed in real-time at inference from a feature store (Redis)
  • LinkedIn (feed ranking): cross-features between user and content (e.g., user_industry × post_topic) as explicit sparse features fed to linear layers — interaction features bridge the gap between linear models and learned embeddings
🎯 Interview Gold

What to say: "Feature engineering is where I spend most of my ML project time — a well-engineered simple model often beats a poorly-featured complex one. My process: (1) understand domain semantics first — what ratios, lags, or interactions should matter physically? (2) create candidate features, (3) evaluate with SHAP + permutation importance — unbiased, shows direction and magnitude. For high-cardinality categoricals: target encoding with CV-in-fold leakage protection, not one-hot. For time series: lag features, rolling aggregations, and Fourier seasonality terms."

Common mistake to avoid saying: "I target-encoded the city column using the full dataset mean." — This is classic leakage. Target encoding must be computed within CV folds (fit on training fold, apply to validation fold).

Expect the follow-up: "How do you handle feature selection when you have 500 features?" — Answer: (1) remove near-zero variance features; (2) remove features with correlation >0.95 (keep one from each correlated cluster); (3) use SHAP importance on a trained model to rank remaining features; (4) iteratively remove bottom-SHAP features and check CV metric — stop when metric drops. This is more principled than RFE and faster than Boruta for >100 features.
