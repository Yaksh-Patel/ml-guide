SQL — Data Science Patterns
🗃️ SQL
-- Running average payment rate (window function)
SELECT customer_id, month,
  AVG(payment_ratio) OVER (PARTITION BY customer_id
    ORDER BY month ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) AS rolling_3m_rate
FROM payments;

-- Cohort retention analysis
SELECT cohort_month, months_since_join,
  COUNT(DISTINCT user_id) AS retained_users,
  COUNT(DISTINCT user_id) * 1.0 / FIRST_VALUE(COUNT(DISTINCT user_id))
    OVER (PARTITION BY cohort_month ORDER BY months_since_join) AS retention_rate
FROM user_cohorts
GROUP BY cohort_month, months_since_join;

-- Customers who defaulted within 6 months of application
SELECT a.customer_id, a.app_date, d.default_date
FROM applications a
LEFT JOIN defaults d ON a.customer_id = d.customer_id
  AND d.default_date BETWEEN a.app_date AND a.app_date + INTERVAL '6 months'
WHERE d.customer_id IS NOT NULL;

Common Mistakes & Misconceptions
  • Using Python loops instead of vectorized pandas/numpy — iterating over DataFrame rows with for loops is 100–1000× slower than vectorized operations. Use df.apply() sparingly; prefer df['col'].map(), vectorized arithmetic, and numpy broadcasting. Profile with %timeit before optimising
  • Writing SQL without window functions for ranking problems — most SQL interview questions about "top N per group" or "previous value" are solved cleanly with window functions (ROW_NUMBER, RANK, LAG, LEAD, SUM OVER). Writing subqueries or joins for these signals unfamiliarity with modern SQL
  • Not using sklearn Pipelines — fitting a scaler on the full dataset then running CV manually leads to leakage. Pipeline(steps=[('scaler', ...), ('model', ...)]) automatically refits preprocessors on each training fold. It also makes code cleaner and serialises the full preprocessing + model as one artifact
  • Reinventing NumPy with loops in ML code — matrix operations, broadcasting, and vectorised comparisons are the core of efficient ML code. A for-loop computing pairwise distances is O(n²) Python overhead; np.einsum or scipy.spatial.distance.cdist achieves the same in C, 100× faster
  • Ignoring time complexity in DS interviews — data science coding interviews increasingly test algorithmic thinking. Know: dict/set lookups are O(1); list.append is O(1) amortised; sorting is O(n log n). Two-pointer and sliding window reduce O(n²) brute-force solutions to O(n)
How It's Used at Top Companies
  • Airbnb / Lyft (data science coding): SQL interview questions heavily test window functions — "find the second-highest revenue day per market" requires RANK() OVER (PARTITION BY market ORDER BY revenue DESC); candidates who write subquery equivalents are marked down for readability
  • Meta (DS interviews): Python coding round tests pandas manipulation (groupby, merge, pivot), statistical reasoning (A/B test analysis), and occasionally LeetCode Medium-level algorithms (sliding window, hashmaps). The pandas portion is where most DS candidates lose points
  • Google (SQL in DS interviews): BigQuery SQL is the standard; complex CTEs, window functions, and ARRAY_AGG / UNNEST for nested structures are tested. Candidates expected to write production-quality SQL, not just basic SELECT/WHERE
  • Production ML engineering: pandas for data processing, SQLAlchemy/DuckDB for feature queries, sklearn Pipelines for clean experiment code — the ability to write clean, efficient, non-leaky Python is a core DS production skill
🎯 Interview Gold

What to say: "For DS coding: pandas over loops always (vectorise — 100× faster), sklearn Pipelines prevent leakage (preprocessors refit on each CV fold automatically), and window functions in SQL (ROW_NUMBER, RANK, LAG, LEAD, SUM OVER PARTITION BY) are essential for ranking and time-series queries. For DSA: sliding window, two pointers, and hashmaps cover 80% of DS interview algorithm questions. Most importantly: communicate your approach before writing code — interviewers care about reasoning and trade-offs, not just correct syntax."

Common mistake to avoid saying: "I'll iterate over the rows to compute this." — Row-level Python loops on DataFrames are an immediate red flag in DS interviews. Vectorised pandas or numpy is always the expected answer for bulk data operations.

Expect the follow-up: "How would you efficiently compute a 7-day rolling average in pandas?" — Answer: df['col'].rolling(window=7, min_periods=1).mean() — O(n) vectorised C implementation. Not a for loop. For time-indexed DataFrames, use .rolling('7D') for calendar-based windows that handle irregular timestamps correctly.
