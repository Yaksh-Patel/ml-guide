Graph Neural Networks
Graphs are the natural structure for social networks, molecules, knowledge bases, fraud rings, and recommendation systems — any domain where relationships between entities matter as much as the entities themselves. GNNs learn representations by iteratively aggregating information from a node's neighbourhood, encoding both node features and graph structure into embeddings used for node classification, link prediction, and graph classification.
Why Graphs Require Special Models
  • Standard ML assumes i.i.d. samples — graph nodes are not independent; they depend on their neighbours
  • Graphs have no fixed input size — a node can have 1 or 10,000 neighbours; standard neural nets can't handle variable-size inputs directly
  • Graphs are permutation invariant — relabelling nodes shouldn't change the output
  • Graph structure itself is informative — a node connected to many high-degree nodes (hubs) is different from one connected to isolated nodes, even if features are identical

Part 1 — The Message Passing Framework
Message Passing Neural Networks (MPNN) — The Unified Framework
Almost every GNN is a special case of message passing. Each layer has two steps: aggregate messages from neighbours, then update the node's own representation.
$$\begin{aligned} \text{Layer } k \text{ — aggregate neighbour messages:} \\ \mathbf{m}_v^{(k)} &= \text{AGGREGATE}^{(k)}\!\left(\left\{\mathbf{h}_u^{(k-1)} : u \in \mathcal{N}(v)\right\}\right) \\ \text{Update node representation:} \\ \mathbf{h}_v^{(k)} &= \text{UPDATE}^{(k)}\!\left(\mathbf{h}_v^{(k-1)},\ \mathbf{m}_v^{(k)}\right) \\ \text{Final embedding after } K \text{ layers:} \\ \mathbf{z}_v &= \mathbf{h}_v^{(K)} \quad \text{(encodes } K\text{-hop neighbourhood)} \end{aligned}$$
Key intuition

After 1 layer, each node sees its immediate neighbours. After 2 layers, it sees neighbours-of-neighbours. After K layers, it sees its K-hop neighbourhood. Deeper = larger receptive field — but also more expensive and prone to over-smoothing (all nodes converge to the same representation).

Part 2 — Key GNN Architectures
GCN — Graph Convolutional Network (Kipf & Welling, 2017)
The simplest and most widely cited GNN. Aggregation = normalised sum of neighbour features; no learnable aggregation weights.
$$\begin{aligned} \mathbf{H}^{(k)} &= \sigma\!\left(\tilde{D}^{-\frac{1}{2}}\tilde{A}\tilde{D}^{-\frac{1}{2}}\,\mathbf{H}^{(k-1)}\mathbf{W}^{(k)}\right) \\ \tilde{A} &= A + I \quad \text{(add self-loops so node aggregates itself)} \\ \tilde{D}_{ii} &= \sum_j \tilde{A}_{ij} \quad \text{(degree matrix for normalisation)} \end{aligned}$$
import torch
import torch.nn.functional as F
from torch_geometric.nn import GCNConv

class GCN(torch.nn.Module):
    def __init__(self, in_ch, hid_ch, out_ch):
        super().__init__()
        self.conv1 = GCNConv(in_ch, hid_ch)
        self.conv2 = GCNConv(hid_ch, out_ch)

    def forward(self, x, edge_index):
        # x: [N, in_ch], edge_index: [2, E]
        x = self.conv1(x, edge_index)
        x = F.relu(x)
        x = F.dropout(x, p=0.5, training=self.training)
        x = self.conv2(x, edge_index)
        return F.log_softmax(x, dim=1)

Properties & limitations
  • Symmetric normalisation ($\tilde{D}^{-1/2}\tilde{A}\tilde{D}^{-1/2}$) prevents high-degree nodes from dominating
  • Transductive only — must see all nodes during training (no new nodes at inference)
  • Treats all neighbours equally — no attention mechanism
  • Over-smoothing at depth >3–4 layers — all nodes converge to similar representations
  • Directed graphs require asymmetric normalisation ($\tilde{D}^{-1}\tilde{A}$)
GraphSAGE — Inductive Graph Learning (Hamilton et al., 2017)
Fixes GCN's inductive limitation by learning a general aggregation function rather than memorising fixed embeddings. Can generalise to unseen nodes and graphs — critical for production systems where new users/items arrive continuously.
$$\begin{aligned} \mathbf{h}_{\mathcal{N}(v)}^{(k)} &= \text{AGGREGATE}_k\!\left(\left\{\mathbf{h}_u^{(k-1)},\ \forall u \in \mathcal{N}(v)\right\}\right) \\ \mathbf{h}_v^{(k)} &= \sigma\!\left(\mathbf{W}^{(k)} \cdot \text{CONCAT}\!\left(\mathbf{h}_v^{(k-1)},\ \mathbf{h}_{\mathcal{N}(v)}^{(k)}\right)\right) \\ \mathbf{h}_v^{(k)} &\leftarrow \mathbf{h}_v^{(k)} / \left\|\mathbf{h}_v^{(k)}\right\|_2 \end{aligned}$$
AggregatorFormulaProperties
Mean$\frac{1}{|\mathcal{N}(v)|}\sum_{u \in \mathcal{N}(v)} \mathbf{h}_u$Permutation invariant; efficient; most common
Max pooling$\max_{u \in \mathcal{N}(v)} \sigma(\mathbf{W}\mathbf{h}_u)$Captures most prominent feature; ignores weaker signals
LSTMApply LSTM on random permutation of neighboursMore expressive; not permutation invariant — problematic
Sum$\sum_{u \in \mathcal{N}(v)} \mathbf{h}_u$Degree-aware — nodes with more neighbours get larger aggregations
Neighbourhood sampling

For nodes with thousands of neighbours (hubs), full neighbourhood aggregation is infeasible. GraphSAGE samples a fixed-size neighbourhood at each layer (e.g. 25 at layer 1, 10 at layer 2). This makes mini-batch training practical on million-node graphs.

GAT — Graph Attention Network (Veličković et al., 2018)
Replaces uniform/normalised aggregation with learned attention weights — different neighbours contribute differently based on feature similarity.
$$\begin{aligned} e_{vu} &= \text{LeakyReLU}\!\left(\mathbf{a}^T\!\left[\mathbf{W}\mathbf{h}_v \,\|\, \mathbf{W}\mathbf{h}_u\right]\right) \quad \text{(attention score)} \\ \alpha_{vu} &= \text{softmax}_u(e_{vu}) = \frac{\exp(e_{vu})}{\sum_{k \in \mathcal{N}(v)} \exp(e_{vk})} \\ \mathbf{h}_v^{(k)} &= \sigma\!\left(\sum_{u \in \mathcal{N}(v)} \alpha_{vu} \cdot \mathbf{W}\mathbf{h}_u\right) \end{aligned}$$
  • Multi-head attention: run K independent attention heads, concatenate or average results — same intuition as transformer multi-head attention
  • Attention weights are interpretable — which neighbours does the model weight most?
  • Stronger than GCN on heterophilic graphs (where dissimilar nodes connect)
from torch_geometric.nn import GATConv

class GAT(torch.nn.Module):
    def __init__(self, in_ch, hid_ch, out_ch, heads=8):
        super().__init__()
        self.conv1 = GATConv(in_ch, hid_ch,
                             heads=heads, dropout=0.6)
        self.conv2 = GATConv(hid_ch*heads, out_ch,
                             heads=1, concat=False,
                             dropout=0.6)
    def forward(self, x, edge_index):
        x = F.dropout(x, p=0.6, training=self.training)
        x = F.elu(self.conv1(x, edge_index))
        x = F.dropout(x, p=0.6, training=self.training)
        return self.conv2(x, edge_index)

Part 3 — GNN Tasks & Applications
Node Classification
  • Predict a label for each node using node features + graph structure
  • Examples: classify users as fraudsters/legitimate; predict paper topic in citation network; classify proteins by function
  • Loss: cross-entropy on labelled nodes; gradients flow through the whole graph
  • Semi-supervised: train on small set of labelled nodes, predict on all unlabelled nodes
Link Prediction
  • Predict whether an edge exists between two nodes (or will exist in the future)
  • Examples: friend recommendation, product recommendation, drug-target interaction, knowledge graph completion
  • Score: $\hat{A}_{uv} = \sigma(\mathbf{z}_u^T \mathbf{z}_v)$ — dot product of learned embeddings
  • Trained with negative sampling: sample random non-edges as negatives
Graph Classification
  • Predict a single label for an entire graph
  • Examples: molecule property prediction (toxic/non-toxic), protein function, program bug detection
  • Requires readout / pooling: aggregate all node embeddings into a single graph embedding (sum, mean, hierarchical pooling)
  • GIN is theoretically the most expressive architecture for this task
GNNs for Fraud Detection — A Worked Example
Fraud rings are hard to detect with tabular features alone. But fraudsters share devices, IP addresses, phone numbers, and bank accounts — forming distinctive subgraph patterns that GNNs can learn.
Graph construction
  • Nodes: transactions, accounts, devices, IP addresses
  • Edges: account shares device, transaction shares IP, accounts linked by same phone
  • Node features: transaction amount, account age, location, velocity metrics
  • Labels: confirmed fraud (1), legitimate (0) — highly imbalanced
Why GNN outperforms tabular model
  • Individual fraudulent transaction may look legitimate in isolation (features OK)
  • But the account is connected to 5 other accounts that all received chargebacks — structural signal
  • GNN propagates this neighbourhood signal: node embedding encodes "this account is connected to fraud"
  • Result: 15–30% recall lift over XGBoost on fraud rings (Alibaba, Amazon fraud research)
from torch_geometric.nn import SAGEConv
import torch.nn as nn

class FraudGNN(nn.Module):
    def __init__(self, in_ch, hid_ch=64):
        super().__init__()
        self.sage1 = SAGEConv(in_ch, hid_ch)
        self.sage2 = SAGEConv(hid_ch, hid_ch)
        self.classifier = nn.Linear(hid_ch, 2)

    def forward(self, x, edge_index):
        x = F.relu(self.sage1(x, edge_index))
        x = F.dropout(x, p=0.3, training=self.training)
        x = F.relu(self.sage2(x, edge_index))
        return self.classifier(x)  # node-level logits

GNNs for Recommender Systems — LightGCN & PinSage
LightGCN (He et al., 2020)
$$\begin{aligned} \mathbf{e}_u^{(k+1)} &= \sum_{i \in \mathcal{N}_u} \frac{1}{\sqrt{|\mathcal{N}_u||\mathcal{N}_i|}} \mathbf{e}_i^{(k)} \\ \mathbf{e}_u^* &= \frac{1}{K+1}\sum_{k=0}^{K} \mathbf{e}_u^{(k)} \quad \text{(layer combination)} \\ \hat{y}_{ui} &= {\mathbf{e}_u^*}^T \mathbf{e}_i^* \quad \text{(predicted score)} \end{aligned}$$
  • Strips GCN to the essentials — no feature transformation, no activation functions
  • State-of-the-art on collaborative filtering; outperforms MF, NGCF on most benchmarks
  • Trained with BPR loss: $\mathcal{L} = -\sum \ln\sigma(\hat{y}_{ui} - \hat{y}_{uj})$ (observed item scored higher than unobserved)
PinSage (Ying et al., Pinterest 2018)
  • GraphSAGE on Pinterest's pin-board bipartite graph (3B nodes, 18B edges)
  • Importance-based neighbourhood sampling using random walks — sample neighbours proportionally to visit frequency, not uniformly
  • Curriculum training: hard negatives (semantically similar but non-interacted) added later in training
  • Deployed for "Related Pins" — one of the highest-impact recommendation improvements at Pinterest
Bipartite Graphs — Foundation of Collaborative Filtering
The user-item interaction graph is a bipartite graph: nodes split into users U and items V, edges only connect U↔V. All collaborative filtering methods — matrix factorisation, LightGCN, PinSage — operate on this structure.
$$\begin{aligned} G &= (U \cup V,\ E), \quad E \subseteq U \times V \\ R[u,v] &= \text{interaction strength (rating, click, purchase)} \\ R &\approx \mathbf{P}\mathbf{Q}^T, \quad \mathbf{P} \in \mathbb{R}^{|U|\times k},\ \mathbf{Q} \in \mathbb{R}^{|V|\times k} \\ \hat{R}[u,v] &= \mathbf{p}_u^T \mathbf{q}_v \quad \text{(matrix factorisation prediction)} \end{aligned}$$

Training GNNs at Scale — Key Challenges
The neighbour explosion problem
  • Full-batch GNN on a million-node graph requires the entire graph in GPU memory — infeasible
  • Mini-batch training: sample target nodes → expand to K-hop neighbourhoods → exponential blow-up
  • Solutions: neighbourhood sampling (GraphSAGE), cluster-GCN (partition graph into clusters, train on subgraphs), GraphSAINT (importance-based subgraph sampling)
Over-smoothing
  • With many GNN layers, all node embeddings converge to the same value — model loses discriminative power
  • Fix: use 2–3 layers (sweet spot for most tasks); residual connections (jumping knowledge networks); PairNorm regularisation; DropEdge (randomly drop edges during training)
  • In practice: 2-layer GNNs often match or outperform deeper ones
Common Mistakes & Misconceptions
  • Using GCN for inductive tasks — GCN is transductive (requires all nodes during training). For new users/items arriving at inference, use GraphSAGE or GAT
  • Too many layers — over-smoothing sets in fast. Start with 2 layers; only add more if validation performance improves
  • Ignoring negative sampling strategy — for link prediction, random negative sampling is easy but inefficient. Hard negatives (plausible but non-existent edges) are critical for good embeddings
  • Treating the graph as fixed — in production, graphs evolve: new nodes, new edges, deleted edges. Design the pipeline for incremental updates; GraphSAGE handles this well
  • Not accounting for class imbalance on node labels — fraud, spam, and churn are rare events. Use weighted cross-entropy or focal loss; oversample fraudulent subgraphs
  • Leaking future edges into training — for link prediction on temporal graphs, always use edges before a cutoff for training; future edges for test. Temporal data leakage is a common bug

🎯 Interview Gold

What is a GNN? "A GNN learns node embeddings by iteratively aggregating information from a node's neighbourhood — after K layers, each node's embedding encodes its K-hop neighbourhood structure. The core operation is message passing: each node receives messages from its neighbours, aggregates them, and updates its own representation. Different GNNs differ in how they aggregate (uniform sum in GCN, attention-weighted in GAT, sampled in GraphSAGE)."

GCN vs GraphSAGE? "GCN is transductive — it learns embeddings for a fixed graph and can't generalise to unseen nodes. GraphSAGE is inductive — it learns a general aggregation function that can be applied to any node's neighbourhood, including new nodes added after training. For production where new users/items arrive constantly, GraphSAGE is the right choice."

Why GNNs for fraud detection? "Individual fraudulent transactions often look legitimate in isolation. But fraud rings share devices, IPs, and accounts — forming distinctive subgraph patterns. A GNN propagates signals across the graph: if an account connects to five others that received chargebacks, that structural signal gets encoded into the account's embedding even if its own features look clean. This catches coordinated fraud that tabular models miss entirely."

What is over-smoothing? "With many GNN layers, each node aggregates from an increasingly large neighbourhood — eventually every node sees the whole graph and all embeddings converge to the same value. The fix: use 2–3 layers (not 10), add residual connections, or use jumping knowledge networks that combine representations from all layers."
