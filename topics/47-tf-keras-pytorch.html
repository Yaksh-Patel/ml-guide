TensorFlow 2.x — Core Patterns
🗃️ SQL
import tensorflow as tf
from tensorflow import keras

# ─── 1. SEQUENTIAL API (simple) ───────────────────────────────
model = keras.Sequential([
    keras.layers.Input(shape=(20,)),
    keras.layers.Dense(64, activation='relu'),
    keras.layers.BatchNormalization(),
    keras.layers.Dropout(0.3),
    keras.layers.Dense(32, activation='relu'),
    keras.layers.Dense(1, activation='sigmoid')
])

# ─── 2. FUNCTIONAL API (for complex graphs) ───────────────────
inputs = keras.Input(shape=(20,))
x = keras.layers.Dense(64, activation='relu')(inputs)
x = keras.layers.BatchNormalization()(x)
x = keras.layers.Dropout(0.3)(x)
outputs = keras.layers.Dense(1, activation='sigmoid')(x)
model = keras.Model(inputs, outputs)

# ─── 3. COMPILE + TRAIN ───────────────────────────────────────
model.compile(
    optimizer=keras.optimizers.AdamW(learning_rate=1e-3, weight_decay=1e-4),
    loss='binary_crossentropy',
    metrics=['AUC', 'Precision', 'Recall']
)

callbacks = [
    keras.callbacks.EarlyStopping(patience=5, restore_best_weights=True),
    keras.callbacks.ReduceLROnPlateau(factor=0.5, patience=3),
    keras.callbacks.ModelCheckpoint('best_model.keras', save_best_only=True)
]

history = model.fit(
    X_train, y_train,
    validation_split=0.2,
    epochs=100,
    batch_size=64,
    callbacks=callbacks
)

# ─── 4. SUBCLASSED MODEL (custom training loop) ───────────────
class CustomModel(keras.Model):
    def __init__(self):
        super().__init__()
        self.dense1 = keras.layers.Dense(64, activation='relu')
        self.out = keras.layers.Dense(1, activation='sigmoid')
    def call(self, x, training=False):
        return self.out(self.dense1(x))
    def train_step(self, data):
        X, y = data
        with tf.GradientTape() as tape:
            y_pred = self(X, training=True)
            loss = self.compute_loss(y=y, y_pred=y_pred)
        grads = tape.gradient(loss, self.trainable_variables)
        self.optimizer.apply_gradients(zip(grads, self.trainable_variables))
        return {m.name: m.result() for m in self.metrics}

# ─── 5. SAVE / EXPORT ─────────────────────────────────────────
model.save('model.keras')                   # Keras format
model.export('saved_model/')                # TF SavedModel for serving
# TFLite for mobile:
converter = tf.lite.TFLiteConverter.from_keras_model(model)
tflite_model = converter.convert()

Key Differences Deep Dive
  • Computational graph: PyTorch builds graph dynamically as you execute (eager by default). TF 2 is also eager but @tf.function compiles to static graph for speed — can be hard to debug.
  • Debugging: PyTorch = put a print() or breakpoint anywhere. TF graph mode = must use tf.print() inside the graph.
  • GPU memory: TF claims all GPU memory upfront; set tf.config.experimental.set_memory_growth(gpu, True) to avoid this.
  • DDP training: PyTorch DistributedDataParallel is cleaner API than TF's MirroredStrategy for multi-GPU.
Common Gotchas
  • PyTorch: forget zero_grad() → accumulated gradients. Forget model.eval() → dropout active during inference.
  • TF/Keras: training=True/False controls BN and Dropout — pass it in model(x, training=True). Default model.predict() sets training=False.
  • Data shapes: PyTorch expects [B, C, H, W] (NCHW); TF expects [B, H, W, C] (NHWC) — a common transposition bug.
  • Loss functions: PyTorch BCEWithLogitsLoss (logits) ≠ BCELoss (probs). Use the logits version — numerically stable.
Common Mistakes & Misconceptions
  • Forgetting optimizer.zero_grad() in PyTorch — PyTorch accumulates gradients by default. Without zero_grad() before each backward pass, gradients from previous batches add to current ones, effectively using an ever-growing batch size. This is a silent bug — no error thrown, just incorrect updates
  • Not switching model.eval() / model.train() — BatchNorm and Dropout behave differently in training vs inference mode. model.eval() disables dropout and uses running statistics for BatchNorm. model.train() re-enables both. Always switch modes explicitly; forgetting model.eval() at inference causes noisy, inconsistent predictions
  • Using BCELoss instead of BCEWithLogitsLoss — BCELoss expects probabilities (after sigmoid); BCEWithLogitsLoss expects raw logits and applies sigmoid internally using the log-sum-exp trick for numerical stability. Always use BCEWithLogitsLoss — it's more stable and avoids log(0) issues with extreme logits
  • Tensor shape mismatches from NCHW vs NHWC — PyTorch uses [Batch, Channels, Height, Width]; TensorFlow uses [Batch, Height, Width, Channels]. Loading a PyTorch-trained image and passing it to a TF model (or vice versa) without transposing channels produces silently wrong results, not an error
  • "torch.no_grad() is only needed for inference" — it's also valuable during validation loops. Without no_grad(), PyTorch builds a computation graph for every validation forward pass, wasting memory. Always wrap validation code in with torch.no_grad(): or @torch.inference_mode()
How It's Used at Top Companies
  • Meta (PyTorch — research and production): PyTorch is Meta's internal ML framework; developed it from Caffe2/Torch; TorchScript and torch.compile() bridge research (dynamic graphs) and production (optimised static graphs for TorchServe or ONNX export)
  • Google (TensorFlow/Keras — production serving): TF Serving, TFLite (mobile), TF.js (browser) form the strongest production deployment ecosystem; TPU training via JAX/TF is the primary path for frontier model training at Google
  • Hugging Face (PyTorch-first): the transformers library is PyTorch-native; all BERT, GPT, LLaMA implementations are in PyTorch; Keras 3 compatibility added later. PyTorch dominates the NLP/LLM research and production community
  • NVIDIA (TensorRT): model optimisation for GPU inference — takes PyTorch/ONNX models and compiles them to optimised TensorRT engines with FP16/INT8 quantisation; standard for low-latency production vision and NLP serving on NVIDIA GPUs
🎯 Interview Gold

What to say: "PyTorch is my default for anything using HuggingFace models or requiring research flexibility — dynamic computation graphs let you insert Python debugging, inspect tensors, and conditionally execute graph branches. TensorFlow's advantage is the production ecosystem: TF Serving, TFLite, TF.js. Keras 3 abstracts over both backends. Four PyTorch gotchas everyone must know: zero_grad() every batch (gradients accumulate); model.eval() at inference (disables dropout/BatchNorm training mode); BCEWithLogitsLoss not BCELoss (numerical stability); torch.no_grad() during validation (skip graph construction)."

Common mistake to avoid saying: "I use model.predict() for inference in PyTorch." — PyTorch doesn't have a predict() method; that's Keras/sklearn. PyTorch inference: model.eval(), then with torch.no_grad(): output = model(x). Confusing these frameworks signals shallow hands-on experience.

Expect the follow-up: "How does torch.compile() improve performance?" — Answer: torch.compile() (introduced in PyTorch 2.0) traces the model's computation graph and compiles it to optimised kernels using TorchInductor (targeting CUDA or CPU). It fuses operations (e.g., attention + softmax + dropout into one kernel), eliminates Python overhead in the training loop, and can achieve 30–50% speedup on transformer training with a single line: model = torch.compile(model).
