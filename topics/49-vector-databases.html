Vector Databases
The infrastructure layer behind every RAG system, semantic search, and embedding-based recommendation. Vector databases efficiently store, index, and retrieve high-dimensional embedding vectors using approximate nearest-neighbour search — finding the most semantically similar items in milliseconds across billions of vectors.
Why Vector Databases Exist
Traditional databases query on exact values. Vector databases query on semantic similarity — "find the 10 documents most similar in meaning to this query." Brute-force exact similarity search is O(n·d) for n vectors of dimension d — completely infeasible at scale.
  • Dense retrieval: text/image/audio → embedding → similarity search → ranked results
  • Use cases: RAG knowledge retrieval, semantic search, deduplication, recommendation, anomaly detection, face recognition
  • Scale challenge: OpenAI text-embedding-3-large produces 3072-dim vectors; a 10M document corpus = 30B floats = 120GB at float32
Similarity Metrics
MetricFormulaWhen to use
Cosine similarity$\frac{u \cdot v}{\|u\|\|v\|}$Text embeddings — magnitude irrelevant, direction matters
Dot product$u \cdot v$When embeddings are L2-normalised (same as cosine); also for MRL embeddings
Euclidean (L2)$\|u - v\|_2$Image embeddings, spatial data, when magnitude matters
Manhattan (L1)$\|u - v\|_1$Sparse high-dimensional data; robust to outliers
Tip

Always L2-normalise embeddings before indexing if using cosine similarity — cosine becomes equivalent to dot product, which is faster to compute and FAISS/HNSW optimise for it.

Approximate Nearest Neighbour (ANN) Algorithms
HNSW — Hierarchical Navigable Small World
The dominant ANN algorithm in production. Builds a multi-layer graph where the top layers are sparse (long-range navigation) and bottom layers are dense (local search). Query traverses top to bottom, pruning the search space at each layer.
Key parameters
ParameterEffectTypical value
MMax edges per node per layer — higher = better recall, more memory16–64
ef_constructionSearch width during index build — higher = better graph quality, slower build100–400
ef_searchSearch width at query time — tune to hit recall target50–200
Performance characteristics
  • Query speed: O(log n) — practically 1–10ms for millions of vectors
  • Recall@10: 95–99% achievable with tuned ef_search
  • Memory: ~(4·d + M·8) bytes/vector — heavier than flat index
  • No training required — incremental inserts supported
  • Weakness: can't do filtered search natively (workaround: post-filter or HNSW with metadata)
IVF — Inverted File Index
Clusters vectors into Voronoi cells via k-means. At query time, probe only the nearest k clusters instead of all vectors.
IVF index: cluster vectors into nlist centroids Query: find nearest nprobe centroids, search only those cells Recall vs speed tradeoff: nprobe=1 (fast, low recall) → nprobe=nlist (exact, slow)
  • Requires training (k-means on representative sample — typically 30–300× nlist vectors)
  • Memory-efficient — scales to billions of vectors on GPU (FAISS-GPU)
  • Combined with PQ (Product Quantisation): IVF+PQ compresses vectors by 8–32× with ~5% recall drop

FAISS — Facebook AI Similarity Search
The most widely used ANN library. CPU and GPU support, multiple index types, and a composable index factory string. Used internally by Pinecone, Weaviate, and most vector DBs.
import faiss, numpy as np

d = 1536  # OpenAI ada-002 dimension
n = 100_000

# Build HNSW index (M=32, ef=200)
index = faiss.IndexHNSWFlat(d, 32)
index.hnsw.efConstruction = 200
index.hnsw.efSearch = 100

# Normalise + add vectors
vecs = np.random.randn(n, d).astype('float32')
faiss.normalize_L2(vecs)
index.add(vecs)

# Query: top-5 nearest neighbours
query = np.random.randn(1, d).astype('float32')
faiss.normalize_L2(query)
distances, indices = index.search(query, k=5)

# IVF+PQ for large-scale (billion vectors)
# "IVF4096,PQ64" = 4096 clusters, 64 PQ subquantisers
index2 = faiss.index_factory(d, "IVF4096,PQ64")
index2.train(vecs)   # requires training
index2.add(vecs)
index2.nprobe = 32   # probe 32 of 4096 clusters

Embedding Pipeline Design
End-to-End Embedding Pipeline
1. Embed
  • Choose embedding model based on task: text, code, multimodal, domain-specific
  • Batch embed documents offline (OpenAI batch API, or local model with GPU)
  • Dimension: 768–3072 for text; higher = better quality, more storage
  • Always L2-normalise before inserting if using cosine similarity
2. Chunk & Store
  • Chunk documents before embedding — embedding model has a context window (usually 512–8192 tokens)
  • Store original text + metadata alongside each vector (chunk ID, source URL, timestamp)
  • Choose index type: HNSW for recall, IVF+PQ for billion-scale, pgvector for SQL integration
3. Query & Retrieve
  • Embed the query with the same model — embedding model mismatches destroy recall
  • ANN search returns top-k candidate IDs + distances
  • Apply metadata filters (date range, source, category) — pre-filter or post-filter
  • Fetch full text for top-k; pass to LLM as context

Filtered Vector Search — A Critical Production Concern
Most real applications require filtered search: "find the 10 most similar documents that are also from the last 30 days and in category X." Naive approaches break recall or performance.
Pre-filtering (filter-then-search)
  • Apply metadata filter first → search only matching subset
  • Problem: if filter is selective, HNSW graph becomes disconnected — recall collapses
  • Pinecone's approach: namespace partitioning + dynamic ef_search boost
Post-filtering (search-then-filter)
  • Retrieve top-K (K >> k), then apply metadata filter to the candidates
  • Problem: may return fewer than k results if filter is strict
  • Solution: over-fetch (K=10×k), accept occasional shortfall
Best practice

Qdrant's payload index + HNSW with filtered search is currently the best OSS option — applies filters during graph traversal, maintaining recall without over-fetching. Weaviate's GraphQL + hybrid also handles this well.

Common Mistakes & Misconceptions
  • Using different models for indexing vs querying — destroys the embedding space; recall drops to near-zero. Always use the exact same model and preprocessing
  • Not normalising embeddings — cosine similarity computed on unnormalised vectors gives wrong results; always L2-normalise before inserting
  • Assuming ANN = exact — ANN trades a small recall loss for large speed gains. Always measure recall@k vs a Flat (exact) index baseline
  • Choosing database before benchmarking — recall@10, QPS, and memory vary significantly by data distribution. Always benchmark on your actual vectors
  • Ignoring embedding model dimension vs quality tradeoff — larger dimensions aren't always better; Matryoshka embeddings can match larger dims at lower cost
  • Index size vs nlist mismatch in IVF — rule of thumb: nlist ≈ 4√n (e.g. 4096 for 1M vectors); too few clusters = poor partitioning, too many = training failure

🎯 Interview Gold

How vector search works: "We embed documents and queries into a shared high-dimensional space using the same model. At query time we embed the query and run approximate nearest-neighbour search — typically HNSW — against the indexed document vectors. HNSW builds a multi-layer navigable graph and traverses it top-down in O(log n), achieving 95–99% recall at <5ms for millions of vectors."

ANN vs exact search: "Brute-force exact search is O(n·d) — infeasible at scale. HNSW trades 1–5% recall for 100–1000× speedup. In practice: measure recall@10 vs a Flat index on your specific data. If recall is sufficient (usually >95%), ANN is the clear choice."

Index selection guide: "HNSW for <50M vectors where RAM is available and recall matters. IVF+PQ for billion-scale where memory is constrained — 32× compression with ~5% recall loss. pgvector for teams already on Postgres who need <10M vectors without new infra. Pinecone/Qdrant for managed production with metadata filtering."

Common follow-up — filtered search: "Filtered ANN is non-trivial — pre-filtering with HNSW on selective filters breaks graph connectivity and collapses recall. Best approach: Qdrant's payload-indexed HNSW that applies filters during graph traversal, or over-fetch with post-filtering. Always benchmark recall on your actual filter distributions."
