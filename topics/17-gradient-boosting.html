Gradient Boosting
Sequential ensemble where each tree corrects the residual errors of all previous trees. Reduces bias (unlike Random Forest which reduces variance). The dominant algorithm for structured data competitions.
Core Algorithm
$$\begin{aligned} F_0(x) = \text{base prediction (e.g., mean of } y) \\ r_{im} = -\frac{\partial L(y_i, F_{m-1}(x_i))}{\partial F_{m-1}(x_i)} \quad \text{[pseudo-residuals]} \\ \text{Fit tree } h_m \text{ to pseudo-residuals } \{r_{im}\} \\ F_m(x) = F_{m-1}(x) + \eta \cdot h_m(x) \\ F(x) = F_0 + \eta \cdot \sum_m h_m(x) \end{aligned}$$
For squared loss: pseudo-residuals = actual − predicted (simple residuals). For log-loss: pseudo-residuals = y − σ(F(x)). Each tree is a weak learner (shallow, high bias) trained on the gradient direction of the loss.
XGBoost
  • Second-order Taylor expansion for faster convergence
  • Built-in L1/L2 regularization on leaf weights
  • Handles missing values natively (learns default direction)
  • Parallel tree construction per level
Gold standard
LightGBM
  • Leaf-wise growth (vs level-wise in XGBoost) → deeper, more accurate
  • GOSS: only large-gradient samples for split finding
  • EFB: bundles mutually exclusive features
  • 10-20x faster than XGBoost on large data
Speed king
CatBoost
  • Native categorical encoding (ordered target encoding)
  • Ordered boosting prevents target leakage
  • Symmetric trees → fast inference
  • Minimal hyperparameter tuning needed
Best for cats

📖 The Consulting Analogy

XGBoost hires 500 consultants sequentially. Consultant 1 gives a report. Consultant 2 reads report 1 and specifically studies its mistakes, then corrects them. Consultant 3 corrects what 1 and 2 missed together. Each expert focuses only on the residual errors of all predecessors. By the end, the team has collectively covered every blind spot. Too many rounds? The last consultants start memorizing trivia (overfitting) — early stopping saves you.

Each round, residuals shrink by factor η. After many rounds, F(x) converges to y. Learning rate η < 1 prevents overshoot — you take a fraction of each correction, allowing later trees to fine-tune rather than overcorrect.
Common Mistakes & Misconceptions
  • Not using early stopping — setting n_estimators=1000 and hoping for the best. Always pair with eval_set + early_stopping_rounds; let the data decide when to stop
  • Tuning n_estimators via grid search — it's redundant with early stopping and expensive. Fix n_estimators large, tune other params, let early stopping find the right tree count
  • Confusing boosting and bagging — boosting is sequential (reduces bias), bagging is parallel (reduces variance). Gradient boosting of deep trees reduces both
  • Leaking the validation set into early stopping — if the eval_set is used to select any preprocessing parameters (e.g., imputation, scaling) it's leaked. Put all preprocessing inside a Pipeline
  • Assuming LightGBM always wins — on small datasets (<10k rows) XGBoost is often comparable; CatBoost is better when raw categoricals dominate
  • Ignoring scale_pos_weight for imbalance — set scale_pos_weight = n_negative / n_positive in XGBoost for imbalanced classification; it scales the gradient of positive examples
How It's Used at Top Companies
  • Meta (Ads ranking): LightGBM for click-through rate prediction on hundreds of sparse categorical features; inference latency under 10ms per request at billions of daily calls
  • Stripe (Fraud detection): XGBoost trained on transaction graph features; early stopping with PR-AUC on a held-out time slice to prevent temporal leakage
  • Booking.com: gradient boosting for price ranking and conversion prediction; heavily regularized (lambda, subsample) to prevent overfitting on sparse user history
  • Kaggle competitions: XGBoost / LightGBM win the vast majority of structured-data competitions — the de facto first choice before neural nets
  • Credit scoring industry: XGBoost with monotone constraints (feature must have monotone effect on score) for regulatory compliance — ensures income always positively correlates with creditworthiness
🎯 Interview Gold

What to say: "Gradient boosting fits trees sequentially, each one targeting the negative gradient (pseudo-residuals) of the loss. Unlike Random Forest which reduces variance via bagging, boosting primarily reduces bias. The learning rate η shrinks each tree's contribution — small η needs more trees but generalises better. I always use early stopping: set n_estimators=5000 and let the eval metric on a hold-out determine the optimal round. LightGBM is my default — leaf-wise growth and GOSS make it 10–20× faster than XGBoost on large data with comparable accuracy."

Common mistake to avoid saying: "I tune n_estimators in a grid search." — This signals you don't know early stopping. Always say early stopping instead.

Expect the follow-up: "How does XGBoost differ from vanilla gradient boosting?" — Answer: second-order Taylor approximation of the loss (uses curvature, not just gradient), built-in L1/L2 regularization on leaf weights, and column/row subsampling per tree. These together make it more regularized and faster to converge than sklearn GradientBoostingClassifier.
