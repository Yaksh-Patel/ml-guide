Tokenization & Embeddings
Before any model can process text, it must be converted to vectors. The tokenization and embedding strategy profoundly affects what the model can learn.

BPE: starts with characters, merges most frequent adjacent pairs repeatedly. "unhappiness" → ["un", "happy", "ness"]. Handles unknown words gracefully as subwords.
Word Embeddings
  • Word2Vec: predict surrounding words (CBOW) or center word from context (Skip-gram). Same word → same embedding regardless of context.
  • GloVe: factorize word co-occurrence matrix. Captures global statistics.
  • FastText: extends Word2Vec with character n-grams — handles OOV words and morphology.
  • Contextual (BERT etc.): same word gets different embeddings in different contexts. "bank" in "river bank" vs "savings bank" → different vectors.
Embedding Space Properties
$$\begin{aligned} king - man + woman \approx queen \quad \text{[Word2Vec analogy]} \\ cosine_similarity(Paris, France) \approx cosine_similarity(Berlin, Germany) \\ Nearest neighbors of "cat" \approx_{dog,kitten,feline,pet} \end{aligned}$$
Embeddings encode semantic relationships. Arithmetic in embedding space reflects semantic relationships — the famous "king - man + woman = queen" property. This emerges purely from co-occurrence statistics, not explicit semantic rules.

Common Mistakes & Misconceptions
  • Using [CLS] token as a sentence embedding without fine-tuning — BERT's [CLS] token is only meaningful as a sentence representation after task-specific fine-tuning (it was trained on NSP, not semantic similarity). For off-the-shelf sentence embeddings, use mean pooling of all token hidden states or SBERT — empirically much better than raw [CLS]
  • Comparing embeddings from different models — embedding spaces are model-specific and not comparable. Cosine similarity between a BERT embedding and an OpenAI embedding is meaningless. Always embed queries and documents with the same model
  • Ignoring tokenization boundary effects — BPE tokenises "unhappiness" as ["un", "happiness"] but "unhappy" as ["unhappy"] — subtle changes in word form can produce very different token sequences. For tasks sensitive to character-level patterns (NER, dates, numbers), verify tokenization before assuming the model handles them well
  • "More dimensions = better embedding" — larger embedding dimensions capture more information but increase storage and retrieval cost. Matryoshka Representation Learning (MRL, used in OpenAI text-embedding-3) trains embeddings that can be truncated to any dimension while retaining quality — you can use 256-dim instead of 3072-dim at modest quality cost
  • Not normalising embeddings before cosine similarity — cosine similarity = dot product of unit-normalised vectors. If vectors are not L2-normalised, dot product and cosine similarity diverge. Always normalise embeddings before using them in ANN indexes or cosine comparisons
How It's Used at Top Companies
  • OpenAI (text-embedding-3): MRL-trained embeddings usable at any dimension from 256 to 3072; the standard embedding model for RAG and semantic search in the OpenAI ecosystem; internally fine-tuned on massive retrieval datasets
  • Google (Universal Sentence Encoder / Gecko): dual-encoder architecture trained on massive web data for semantic similarity; the embedding backbone behind Google's passage retrieval in Search and Bard/Gemini RAG
  • Spotify (podcast search): SBERT fine-tuned on podcast transcripts for semantic search; user query embedded at search time, episode embeddings pre-computed and indexed with FAISS — retrieval under 5ms at scale
  • Airbnb (listing search): dual-encoder model trained on (query, clicked listing) pairs; listing embeddings computed offline and indexed; query embedding computed at search time and used for ANN retrieval — production semantic search system serving millions of queries/day
🎯 Interview Gold

What to say: "BPE tokenization handles OOV gracefully — 'ChatGPT' tokenises as ['Chat','G','PT'] and the model works with it. Word2Vec's limitation is polysemy — 'bank' gets one static embedding regardless of context. BERT's contextual embeddings fix this but are expensive to compute. For semantic search in production: SBERT (or BGE-M3 for multilingual) + FAISS for ANN retrieval — pre-compute document embeddings offline, embed queries at inference time, retrieve top-k in <10ms. Always L2-normalise embeddings before cosine comparison."

Common mistake to avoid saying: "I used the [CLS] token embedding from BERT for sentence similarity." — [CLS] without fine-tuning is a poor sentence representation. SBERT's mean-pooled embeddings consistently outperform raw [CLS] for semantic similarity — a well-known empirical finding.

Expect the follow-up: "How does BPE tokenization work?" — Answer: start with character vocabulary. Repeatedly find the most frequent adjacent byte pair and merge it into a new token. Repeat for V iterations (V = vocabulary size target). Result: common words become single tokens; rare words are split into frequent subword pieces. This gives good coverage without OOV and keeps vocabulary size manageable (~50K tokens for most LLMs).
