Data Engineering for ML
Most ML projects fail not because of bad models but because of bad data pipelines. Data engineering for ML covers the full journey from raw data to features ready for training — batch and streaming pipelines, transformation frameworks (Spark, dbt), table formats (Delta Lake, Iceberg), data quality, and the feature engineering infrastructure that keeps training and serving in sync.
Why Data Engineering Is an ML Bottleneck
  • Data scientists spend 80% of time on data — finding, cleaning, transforming, validating — and only 20% on modelling
  • Training-serving skew: features computed differently in training pipelines vs inference APIs causes silent model degradation — the most common production ML failure
  • Reproducibility: without versioned data and deterministic pipelines, experiments can't be reproduced and bugs can't be traced
  • Scale: pandas works for millions of rows; real training data is often hundreds of billions of rows requiring distributed compute

Part 1 — Apache Spark for ML Data Pipelines
Spark — Distributed Data Processing
Spark is the dominant framework for processing data too large for pandas. It distributes computation across a cluster, processes data in-memory (vs Hadoop MapReduce's disk-based approach), and provides DataFrame, SQL, and ML APIs.
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.window import Window

spark = SparkSession.builder \
    .appName("MLFeaturePipeline") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

# Read 100M+ row parquet from S3
df = spark.read.parquet("s3://bucket/transactions/")

# Compute rolling 30-day payment rate per customer
w = Window.partitionBy("customer_id") \
          .orderBy("txn_date") \
          .rangeBetween(-30*86400, 0)  # 30 days in seconds

df_features = df.withColumn(
    "paid_30d", F.sum("is_paid").over(w)
).withColumn(
    "txns_30d", F.count("*").over(w)
).withColumn(
    "payment_rate_30d",
    F.col("paid_30d") / F.col("txns_30d")
)

# Write training dataset to Delta Lake
df_features.write \
    .format("delta") \
    .mode("overwrite") \
    .save("s3://bucket/features/payment_rate/")
Key Spark concepts for ML
  • Lazy evaluation: transformations are not executed until an action (write, collect, show). Spark builds a DAG of operations and optimises before running
  • Partitioning: data is split into partitions processed in parallel. Key for performance — too few = underutilised, too many = scheduling overhead. Rule: 128MB–1GB per partition
  • Window functions: rolling aggregations (7d/30d/90d rates) are core ML features; Spark window functions are the scalable alternative to pandas groupby + rolling
  • Adaptive Query Execution (AQE): auto-optimises join strategies and partition sizes at runtime — always enable in Spark 3+
  • Broadcast join: when one table is small (<10MB), broadcast it to all workers to avoid expensive shuffle join
Common Spark performance pitfalls

Data skew: one partition has 10× more data than others → one task takes 10× longer, blocking the job. Fix: salting keys, skew hints, or AQE skew join. Shuffle: groupBy + join operations require shuffling data across the network — the most expensive operation. Minimise shuffles; cache intermediate results if reused. UDFs: Python UDFs break Spark's JVM optimisations. Use built-in SQL functions (F.*) wherever possible.

Part 2 — dbt for ML Feature Engineering
dbt — Data Build Tool
dbt transforms raw data in your data warehouse using SQL SELECT statements. It handles dependency ordering, incremental loading, testing, and documentation — everything you'd write boilerplate Python for, dbt handles declaratively.
-- models/features/customer_payment_rate.sql
-- dbt model: computes 30-day payment rate per customer

{{
  config(
    materialized = 'incremental',
    unique_key   = ['customer_id', 'snapshot_date'],
    partition_by = {'field': 'snapshot_date', 'data_type': 'date'}
  )
}}

with base as (
    select
        customer_id,
        snapshot_date,
        count(*) as txns_30d,
        sum(case when status='paid' then 1 else 0 end)
            as paid_30d
    from {{ ref('stg_transactions') }}  -- refs ensure ordering
    where txn_date between
        dateadd('day', -30, snapshot_date) and snapshot_date
    {% if is_incremental() %}
        and snapshot_date > (select max(snapshot_date) from {{ this }})
    {% endif %}
    group by 1, 2
)

select
    customer_id,
    snapshot_date,
    paid_30d / nullif(txns_30d, 0) as payment_rate_30d,
    txns_30d
from base
Why dbt for ML
  • Version-controlled SQL: feature logic is in git — auditable, reviewable, reproducible
  • DAG of dependencies: dbt resolves ref() calls into the correct build order automatically
  • Incremental models: only process new data; don't recompute the whole history on every run — essential for large feature tables
  • Built-in testing: not_null, unique, accepted_values, relationships tests validate every model run
  • dbt docs: auto-generated data lineage graph and data dictionary — "what does this feature mean and where does it come from?"
dbt vs Spark for ML

Use dbt when your data is already in a warehouse (BigQuery, Snowflake, Redshift) and fits in SQL. Use Spark when data lives in a data lake (S3/GCS), is very large (>1TB), requires complex windowed aggregations, or needs Python/MLlib transformations. Most modern stacks use both: Spark for raw ingest + heavy processing, dbt for curated feature engineering inside the warehouse.

# dbt tests in schema.yml — run automatically after every build
models:
  - name: customer_payment_rate
    columns:
      - name: customer_id
        tests:
          - not_null
          - unique  # each customer appears once per snapshot date
      - name: payment_rate_30d
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0
              max_value: 1  # rate must be in [0,1]
Part 3 — Table Formats: Delta Lake & Apache Iceberg
Why Table Formats Matter for ML
Raw parquet files in S3 are immutable and stateless — you can't update a row, roll back, or read consistent snapshots. Delta Lake and Iceberg add ACID transactions, versioning, and time travel on top of parquet, solving critical ML data engineering problems.
Delta Lake (Databricks)
  • ACID transactions: concurrent reads and writes without corruption; no more partial writes breaking downstream jobs
  • Time travel: query any historical version — VERSION AS OF 42 or TIMESTAMP AS OF '2024-01-01'
  • Schema enforcement + evolution: rejects writes that don't match the schema; controlled schema changes
  • DML operations: UPDATE, DELETE, MERGE INTO — update individual rows in parquet (previously impossible)
  • Z-order clustering: co-locate related data on disk for faster queries on common filter columns
from delta.tables import DeltaTable

# Time travel: reproduce training dataset from 3 months ago
df_train = spark.read.format("delta") \
    .option("timestampAsOf", "2024-01-15") \
    .load("s3://bucket/features/payment_rate/")

# Upsert (MERGE): update existing rows, insert new ones
DeltaTable.forPath(spark, "s3://bucket/features/") \
    .alias("target") \
    .merge(new_data.alias("source"),
           "target.customer_id = source.customer_id") \
    .whenMatchedUpdateAll() \
    .whenNotMatchedInsertAll() \
    .execute()
Apache Iceberg (Netflix / Apple)
  • Same core features as Delta Lake: ACID, time travel, schema evolution
  • Hidden partitioning: partitions managed by Iceberg, not exposed to query writers — no need to remember WHERE dt=...
  • Multi-engine support: natively supported by Spark, Flink, Trino, Hive, DuckDB — Delta is more tightly coupled to Spark/Databricks
  • Partition evolution: change partition strategy on existing data without rewriting
  • Row-level deletes: GDPR deletion (right to erasure) without rewriting entire partitions
Delta vs Iceberg

Delta Lake is the default if you're on Databricks or already use Spark heavily. Iceberg is preferred for multi-engine environments (Flink + Spark + Trino) or when hidden partitioning is important. Both are production-ready; the choice is often ecosystem fit rather than capability gap.

Part 4 — Feature Pipelines: Batch vs Streaming
Batch Feature Pipelines
Compute features on a schedule (hourly, daily, weekly) from historical data. Best for features that don't need sub-minute freshness and where history matters.
Typical Airflow DAG pattern 
from airflow import DAG
from airflow.providers.apache.spark.operators.spark_submit import SparkSubmitOperator
from airflow.providers.dbt.cloud.operators.dbt import DbtCloudRunJobOperator

with DAG("ml_feature_pipeline",
         schedule_interval="0 3 * * *",  # 3am daily
         catchup=False) as dag:

    # Step 1: Run Spark job to compute raw aggregations
    spark_agg = SparkSubmitOperator(
        task_id="compute_raw_features",
        application="s3://code/feature_agg.py",
        conf={"spark.executor.memory": "8g"}
    )

    # Step 2: dbt transformation inside warehouse
    dbt_features = DbtCloudRunJobOperator(
        task_id="dbt_feature_models",
        job_id=12345
    )

    # Step 3: Materialise to feature store
    load_feast = PythonOperator(
        task_id="load_to_feature_store",
        python_callable=push_to_feast
    )

    spark_agg >> dbt_features >> load_feast
Point-in-time correct joins — the critical concept
  • When building training data, you must use feature values that were available at the time of the label event — not current values
  • Example: predicting default on a loan application filed Jan 15. The customer's 30-day payment rate feature must be computed as of Jan 15 — not today's value (which might reflect the loan outcome)
  • Using future feature values = target leakage — model looks great in training but fails in production
  • Feature stores (Feast, Tecton) implement point-in-time joins automatically
# Feast point-in-time correct retrieval
training_df = store.get_historical_features(

    features=[
        "customer_features:payment_rate_30d",
        "customer_features:income_to_debt_ratio",
    ]
).to_df()

Streaming Feature Pipelines — Kafka + Flink
For features that must be fresh within seconds — fraud detection velocity signals, real-time session features, live inventory counts. Streaming pipelines consume events as they arrive and maintain continuously updated aggregations.
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment

env = StreamExecutionEnvironment.get_execution_environment()
t_env = StreamTableEnvironment.create(env)

# Consume from Kafka topic
t_env.execute_sql("""
    CREATE TABLE transactions (
        customer_id BIGINT,
        amount      DOUBLE,
        is_fraud    BOOLEAN,
        event_time  TIMESTAMP(3),
        WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
    ) WITH (
        'connector' = 'kafka',
        'topic'     = 'transactions',
        'format'    = 'json'
    )
""")

# Sliding window: count transactions in last 1 hour
t_env.execute_sql("""
    CREATE TABLE velocity_features AS
    SELECT
        customer_id,
        COUNT(*) AS txn_count_1h,
        SUM(amount) AS txn_amount_1h,
        TUMBLE_START(event_time, INTERVAL '1' HOUR) AS window_start
    FROM transactions
    GROUP BY customer_id,
             TUMBLE(event_time, INTERVAL '1' HOUR)
""")
Streaming architecture for ML
  • Kafka: event bus — all raw events (transactions, clicks, logins) published as topics; durable, replayable
  • Flink: stateful stream processor — maintains running aggregations (counts, sums, rates) in memory as events arrive; checkpoints to S3/HDFS for fault tolerance
  • Redis: online feature store — Flink writes computed features to Redis; model inference API reads from Redis in <1ms
  • Lambda architecture: batch pipeline (daily) for accuracy on historical features; streaming pipeline (real-time) for freshness on velocity features — results merged at serving time

Common Mistakes & Misconceptions
  • Training-serving skew — computing features differently in the training pipeline (Spark SQL) vs the serving API (Python). The feature store enforces identical logic; without it, skew is almost inevitable at scale
  • Target leakage via point-in-time errors — using the current value of a feature that was computed after the label event. Always verify temporal ordering; use feature store's point-in-time join API
  • Using pandas for 100M+ rows — pandas loads everything into RAM; use Spark, DuckDB, or Polars for large-scale feature engineering. A Spark job that runs in 10 minutes processes what pandas would take 3 hours on (if it doesn't OOM first)
  • Overwriting parquet partitions instead of merging — concurrent writes to the same partition without a table format (Delta/Iceberg) causes data corruption. Use MERGE INTO instead of overwrite for updates
  • No data quality checks in training pipeline — a silent upstream schema change (column renamed, values rescaled) trains the model on garbage data. Always gate training on data quality validation
  • Not versioning training datasets — if you can't reproduce the exact training dataset 6 months from now, you can't debug why the model behaved the way it did. Use Delta Lake time travel or DVC to version datasets

🎯 Interview Gold

What is training-serving skew? "Training-serving skew is when features are computed differently during training vs at inference time. For example: training uses a Spark job that computes 30-day payment rate with a left join, but the serving API uses a Python function with an inner join — customers with no payments in 30 days get NULL in training but 0.0 at serving. The model sees a completely different distribution in production. A feature store solves this by enforcing a single feature definition used in both contexts."

What is point-in-time correct joining? "When building training data, you must join features that were available at the exact moment of the prediction — not the current value. If I'm predicting loan default on applications filed in January, my features must be the values known in January, not today's values (which might reflect the outcome). Using future feature values is target leakage — models look perfect in training and fail in production. Feature stores implement time-travel joins to prevent this automatically."

Spark vs dbt for feature engineering? "dbt operates inside a SQL warehouse (BigQuery, Snowflake) — great for teams already there, incremental builds, and built-in testing. Spark is for data lake processing (S3/GCS), data too large for a warehouse, or pipelines requiring Python or complex windowed aggregations. In practice, they complement each other: Spark for heavy lifting on raw data, dbt for curated feature engineering inside the warehouse."

Why Delta Lake instead of plain parquet? "Plain parquet is immutable — you can't update rows, and concurrent writes corrupt data. Delta Lake adds ACID transactions (safe concurrent writes), time travel (reproduce any historical snapshot for debugging or reproducing training data), schema enforcement (upstream changes don't silently corrupt downstream), and DML operations (UPDATE, DELETE, MERGE). For ML specifically: time travel is critical for dataset reproducibility, and MERGE is essential for feature store upserts."
