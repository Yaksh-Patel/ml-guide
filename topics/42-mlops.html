Deployment & Monitoring (MLOps)
The gap between a working notebook and a production ML system is enormous. MLOps is the engineering discipline that bridges it — and the reason most ML projects never reach production.

Feature Store — Why It Matters
A feature store solves two problems: consistency between training and serving, and feature reuse across teams.
  • Training-serving skew: features computed differently in batch (training) vs real-time (serving) → model sees different distributions. Feature store enforces the same logic everywhere.
  • Point-in-time correct joins: when training, join features that were available AT THE PREDICTION TIME — not the current value. Prevents future leakage.
  • Reuse: "customer_30day_payment_rate" computed once, used by fraud, collections, and marketing models
A/B Testing in Production
  • Shadow mode: new model runs alongside old, predictions logged but not acted on. Zero risk, but no business metric feedback.
  • Canary release: route 5–10% of traffic to new model. Monitor, scale up if healthy.
  • Champion-challenger: production model = champion, new model = challenger. Compare performance on live traffic.
  • Multi-armed bandit: dynamically allocate more traffic to better-performing variant. Faster than fixed A/B but harder to interpret.
🎯 Interview Gold

"The most common MLOps failures: (1) training-serving skew — feature computation differs between training pipeline and inference API; (2) no monitoring — model silently degrades for months; (3) no rollback mechanism — bad deployment can't be reverted quickly. My production checklist: containerize (Docker), track experiments (MLflow), version data (DVC), monitor with PSI on feature distributions + business KPIs, and always keep the previous model artifact ready for rollback."
