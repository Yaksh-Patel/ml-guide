Deployment & Monitoring (MLOps)
The gap between a working notebook and a production ML system is enormous. MLOps is the engineering discipline that bridges it — covering containerisation, CI/CD, serving infrastructure, cost optimisation, and monitoring. Most ML projects fail here, not in the modelling.

Containerisation with Docker
Container = reproducible, portable unit. Packages model + dependencies + runtime together. The standard unit of ML deployment.
  • Dockerfile layers: base image → system deps → Python deps → model artifacts → entrypoint
  • Multi-stage builds: build stage (large) → runtime stage (slim) — reduces final image size 5–10×
  • CUDA base images: use nvidia/cuda:12.x-runtime for GPU serving; don't include cudnn unless needed
  • Model weight mounting: mount weights as a volume or pull from S3/GCS at startup — don't bake into image (large image → slow CI)
  • Health checks: /health endpoint critical for Kubernetes liveness probes
Kubernetes for ML Serving
Kubernetes ML serving: requests → LoadBalancer → Pods → model servers with HPA scaling
  • Deployment: defines desired Pod spec (image, resources, env vars); rolling updates with zero downtime
  • Service: stable DNS name + load balances across Pods; ClusterIP for internal, LoadBalancer for external
  • HPA: scales Pod count based on CPU/GPU utilisation or custom metrics (requests/sec)
  • Resource requests/limits: always set both — prevents noisy neighbours from starving model Pods
  • GPU node pools: separate node pool with GPU nodes; taint + toleration ensures only GPU Pods schedule there
  • Readiness probe: Kubernetes only routes traffic after probe succeeds — critical for slow model loading
  • ConfigMaps / Secrets: inject model config and API keys without rebuilding images
  • Namespace separation: staging vs production namespaces prevent accidental cross-contamination
Part 2 — CI/CD for ML
ML CI/CD Pipeline
Standard software CI/CD (test → build → deploy) extended to handle data, models, and experiments — not just code.
ML CI/CD pipeline: code commit triggers data validation → training → evaluation → registry → serving
  • GitHub Actions / Jenkins: trigger training jobs on push to main or on schedule
  • Data validation gate: Great Expectations / Pydantic schema checks before training; fail fast if distribution shifts
  • Automated evaluation gate: new model must beat champion on held-out set — else fail the pipeline
  • Model versioning: every training run tagged with git SHA, dataset hash, hyperparams
  • Rollback: re-promote previous model version in registry → redeploy — takes minutes
  • Smoke tests: integration tests on model endpoint (latency, output schema) before prod promotion

Part 4 — LLM Serving: vLLM & Triton
vLLM — PagedAttention
The bottleneck in LLM serving is the KV cache — it grows linearly with sequence length and batch size, causing memory fragmentation and low GPU utilisation.
  • PagedAttention: manages KV cache like OS virtual memory — allocates fixed-size "pages" on demand, no fragmentation
  • Continuous batching: new requests slot in as sequences finish — no waiting for entire batch to complete; 23× higher throughput vs naive batching
  • Tensor parallelism: splits model weights across multiple GPUs (needed for 70B+ models)
  • Speculative decoding: small draft model proposes tokens; vLLM verifies in parallel; 2–3× faster for high-latency models
  • OpenAI-compatible API — drop-in replacement for OpenAI endpoints
NVIDIA Triton Inference Server
NVIDIA's multi-framework serving platform — supports PyTorch, TensorFlow, ONNX, TensorRT in a single server.
  • Ensemble models: chain preprocessing → model → postprocessing in a single inference request
  • Dynamic batching: accumulates requests and batches before forwarding to model
  • TensorRT backend: JIT compiles PyTorch/ONNX to TensorRT → 3–5× faster GPU inference
  • Model repository: hot-reloading — deploy new model version without server restart
  • GRPC + HTTP: supports both protocols; GRPC preferred for high-throughput production
  • Best for: classical ML serving pipelines, multi-model ensembles, non-LLM deep learning

📖 Profiling tip

Profile P99 latency, not P50. P50 hides tail latency spikes that users actually experience. For ML pipelines, the database / feature store lookup is often the bottleneck — not the model itself.

A/B & Deployment Strategies
  • Shadow mode: new model runs alongside old, predictions logged but not acted on — zero business risk, no business metric feedback
  • Canary release: route 5–10% of traffic to new model; monitor; scale up if healthy
  • Champion-challenger: champion = current production; challenger = new model; compare on live traffic
  • Multi-armed bandit: dynamically allocate traffic to better variant; faster than fixed A/B but harder to interpret for statistical significance

Feature Store — Why It Matters
  • Training-serving skew: features computed differently in batch (training) vs real-time (serving) → model sees different distributions at inference time — feature store enforces the same transformation logic everywhere
  • Point-in-time correct joins: when training, join only features available AT the prediction time — prevents future leakage
  • Reuse across teams: "customer_30day_payment_rate" computed once, used by fraud, collections, and marketing models
  • Offline store (S3/DeltaLake) for training; online store (Redis) for <5ms serving latency
🎯 Interview Gold

What to say: "The most common MLOps failures: (1) training-serving skew — feature computation differs between training pipeline and inference API; (2) no monitoring — model silently degrades for months; (3) no rollback mechanism — a bad deployment can't be reverted quickly. My production checklist: containerise with Docker, track experiments with MLflow, monitor with PSI on feature distributions + business KPIs, canary deploy all changes, and always keep the previous model artifact for hot rollback."

Common mistake: Treating MLOps as just "deploying the model." The hard part is the full loop: drift detection → retraining trigger → CI/CD pipeline → evaluation gate → canary deploy → monitor again. Each step needs automation.

Expect this follow-up: "How would you serve a 70B parameter LLM at low latency?" — Tensor parallelism across multiple A100s, vLLM PagedAttention for KV cache efficiency, INT4 quantisation, speculative decoding with a 3B draft model. Expect 200–500ms P50 latency at reasonable cost.
