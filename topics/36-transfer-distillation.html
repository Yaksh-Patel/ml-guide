Transfer Learning & Knowledge Distillation
Two complementary strategies for leveraging existing knowledge — transfer learning adapts pre-trained representations to new tasks, distillation compresses large model knowledge into deployable ones, and PEFT methods (LoRA, QLoRA, adapters) enable fine-tuning billion-parameter models on a single GPU.
Part 1 — Transfer Learning
The Paradigm Shift
Pre-train a large model on a massive task (ImageNet, language modeling, masked token prediction). The model learns rich, general features. Fine-tune this model on your small labeled dataset — the features transfer.
$$\begin{aligned} &\text{Stage 1: Pre-train on large dataset (self-supervised or supervised)} \\ &\text{Stage 2: Replace/add task head (classification, NER, regression, etc.)} \\ &\text{Stage 3: Fine-tune — options:} \\ &\quad\text{(a) Feature extraction: freeze backbone, train only head} \\ &\quad\text{(b) Full fine-tune: unfreeze all layers, small LR} \\ &\quad\text{(c) Partial fine-tune: unfreeze last N layers} \\ &\quad\text{(d) PEFT: add trainable adapters to frozen backbone} \end{aligned}$$
When to Freeze vs Fine-Tune

Catastrophic Forgetting
When fine-tuning on a new task, the model "forgets" pre-trained knowledge. Prevention:
  • Learning rate warmup: start with very small LR, increase gradually
  • Layer-wise LR decay: lower LR for earlier (more general) layers; higher for later layers
  • Elastic weight consolidation (EWC): penalise changing weights important for old tasks
  • LoRA/Adapters: freeze base model entirely — forgetting impossible by design
Part 2 — Parameter-Efficient Fine-Tuning (PEFT)
Why PEFT? The GPU Memory Problem
Full fine-tuning a 7B-parameter model requires storing: model weights (~14GB at FP16) + optimizer states (~56GB for Adam) + gradients (~14GB) + activations (~variable). Total: easily 100GB+ — requires many A100s and large budgets. PEFT methods reduce trainable parameters by 100–10,000× while retaining most of the quality.

LoRA — Low-Rank Adaptation (Deep Dive)
The core idea: weight updates during fine-tuning are low-rank. Instead of updating the full weight matrix W ∈ ℝ^(d×k), decompose the update ΔW into two low-rank matrices.
$$\begin{aligned} W' &= W_\text{frozen} + \Delta W = W + \underbrace{A \cdot B}_{\Delta W} \\ A &\in \mathbb{R}^{d \times r},\; B \in \mathbb{R}^{r \times k},\; r \ll \min(d,k) \\ h &= W' x = Wx + ABx \quad \text{[only A, B are trained]} \\ \text{Params: } &d \cdot k \to r(d+k)\quad \text{e.g. }4096\times 4096 = 16M \to r=16: 131K \text{ (}\mathbf{0.8\%}\text{)} \end{aligned}$$
LoRA: frozen W plus small low-rank update A×B — only A and B are trained
LoRA Hyperparameter Guide
HyperparameterTypical RangeEffect

📖 Rule of thumb

Start with r=16, α=32, target q_proj + v_proj. If quality is lacking, add k_proj + o_proj + FFN layers. If overfitting, reduce r or add dropout.

QLoRA — Quantised LoRA
Fine-tune a model quantised to 4-bit with LoRA adapters in full precision (BF16). Enables fine-tuning 65B+ models on a single 48GB GPU.
  • NF4 quantisation: Normal Float 4 — optimal for normally distributed weights (most LLMs); lower error than INT4
  • Double quantisation: quantise the quantisation constants too — saves ~0.37 bits/param
  • Paged optimisers: offload optimizer states to CPU RAM during memory spikes; prevents OOM during gradient checkpointing
  • Dequantisation: weights dequantised to BF16 for the forward pass through frozen layers; LoRA adapters stay BF16
  • When to use QLoRA vs LoRA: QLoRA if GPU memory is the bottleneck (single GPU, 13B+ model); plain LoRA if you have enough VRAM (faster)
Other PEFT Methods
  • Adapters: insert small bottleneck MLP modules between frozen transformer layers (down-project → nonlinear → up-project). Originally proposed by Houlsby et al. More params than LoRA but slightly more flexible.
  • Prompt tuning: prepend K learnable "soft prompt" tokens to input; only those K × d_model parameters are trained. Works well for very large models (11B+); underperforms LoRA for smaller models.
  • Prefix tuning: learns prefix key-value pairs injected into every attention layer; more expressive than prompt tuning; more params.
  • BitFit: only fine-tune bias terms. Surprisingly effective for classification tasks at minimal cost. Total trainable params ~0.1% of model.
  • IA³: learn scale vectors for keys, values, and FFN; ~10× fewer params than LoRA; good for extreme resource constraints.
Part 3 — Knowledge Distillation
The Core Idea
Instead of training the student on hard labels (0 or 1), train it to match the teacher's soft probability distribution. Soft labels contain richer information — the teacher's confidence across all classes, including "wrong" but related ones.
$$\begin{aligned} p_T(y|x,\tau) &= \text{softmax}(z_T/\tau) \quad [\tau > 1 \text{ smooths distribution}] \\ \mathcal{L} &= \alpha\cdot\mathcal{L}_{CE}(y_\text{hard}, p_S) + (1-\alpha)\cdot\tau^2\cdot\mathcal{L}_{KL}(p_T\|p_S) \\ \tau &= 1\text{: sharp labels} \quad \tau=4\text{: reveals class similarities} \\ \alpha &= 0.3\text{ (typical): balances hard vs soft loss} \end{aligned}$$
Distillation: student learns from both ground truth AND teacher soft output distribution
Why Soft Labels Are Better
Hard label: cat = [1, 0, 0, 0]. Teacher soft label (τ=4): [0.01, 0.72, 0.20, 0.07] — the teacher reveals that "fox" is similar to "cat". The student learns the class similarity structure for free, without ever needing labeled fox-vs-cat examples.
📖 The Brilliant Tutor

The teacher says "this is likely a fox, somewhat like a cat, very unlikely a car." The student learns not just the answer, but the teacher's uncertainty and reasoning — far richer than binary right/wrong labels.

Common Mistakes & Misconceptions
  • Setting LoRA rank r too high — r=16 or r=32 is almost always sufficient; r=256 wastes memory with marginal gain. The whole point of LoRA is a low-rank update: W + ΔW where ΔW = AB, A ∈ ℝ^(d×r), B ∈ ℝ^(r×k). If r approaches min(d,k), you're approaching full fine-tuning
  • Fine-tuning to inject factual knowledge — fine-tuning teaches the model how to behave (style, format, task patterns), not what to know. Facts injected via fine-tuning are unreliable and go stale. Use RAG for knowledge; fine-tune for behaviour
  • Applying LoRA only to attention layers — the default Hugging Face PEFT config targets Q and V projections. Adding LoRA to K, V, Q, and the FFN up/down projections consistently improves quality, especially for instruction following. Use target_modules=["q_proj","v_proj","k_proj","up_proj","down_proj"]
  • Catastrophic forgetting when full fine-tuning — full fine-tuning on a small task-specific dataset overwrites general capabilities. Mitigations: LoRA (only adapters updated), replay (mix original pretraining data), or EWC (elastic weight consolidation — penalises changes to weights important for previous tasks)
  • Using a teacher model that's too large for distillation — knowledge distillation works best when the student has enough capacity to learn the teacher's soft distributions. A 1B student from a 70B teacher often underperforms a well-tuned 3B model trained directly
How It's Used at Top Companies
  • Microsoft (Azure OpenAI fine-tuning): offers GPT-3.5 and GPT-4o fine-tuning via LoRA-based PEFT; customers use it for domain adaptation (legal, medical, code style) with <1000 examples — the go-to enterprise fine-tuning product
  • Google (DistilBERT / MiniLM): distillation reduced BERT-Base (110M params) to DistilBERT (66M params) at 97% BERT benchmark performance and 60% faster inference — production-deployed in latency-sensitive applications where full BERT is too slow
  • Meta (LLaMA fine-tuning ecosystem): QLoRA fine-tuning of LLaMA models is the standard open-source recipe; entire industry of fine-tuned variants (Alpaca, Vicuna, Mistral-Instruct) built on this infrastructure
  • Hugging Face (PEFT library): standardised LoRA, QLoRA, IA³, prefix tuning into a unified API; the de facto production library for PEFT — target_modules, r, lora_alpha, lora_dropout are the four core config parameters every DS should know
  • Anthropic / OpenAI (Constitutional AI / RLHF): fine-tuning pipeline: supervised fine-tuning (SFT) → reward model training → RLHF/DPO alignment. Each stage is a targeted fine-tuning step with different objectives and data types
🎯 Interview Gold

What to say: "Transfer learning is the highest-leverage technique in applied deep learning. For NLP I start with a pretrained BERT/RoBERTa or an instruction-tuned LLM, add a classification head, and fine-tune with LoRA — I can adapt a 7B LLM to a custom task on a single GPU by training only 0.1% of parameters. The intuition: lower layers learn general features (syntax, semantics), upper layers learn task-specific patterns — so for domain adaptation, only upper layers need significant updating. QLoRA extends this to 13B+ models by quantising the frozen base to 4-bit NF4 while keeping LoRA adapters in BF16 — ~70% VRAM savings at ~2% quality cost."

Common mistake to avoid saying: "I fine-tuned the model to teach it new facts." — Fine-tuning encodes behavioural patterns, not reliable factual knowledge. Facts change and fine-tuned knowledge becomes stale. Use RAG for dynamic knowledge; fine-tune only for style, format, and task-specific reasoning patterns.

Expect the follow-up: "What is the difference between LoRA and full fine-tuning?" — Answer: full fine-tuning updates all parameters (billions of gradients, requires huge GPU memory). LoRA freezes the pretrained weights and adds low-rank update matrices ΔW = AB where rank r ≪ min(d,k). Only A and B are trained — typically 0.1–1% of original parameters. The pretrained weights are unchanged so general capabilities are preserved; the adapters capture task-specific adjustments. LoRA can be merged at inference (W + AB) so there's zero latency overhead.
