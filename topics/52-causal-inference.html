Causal Inference & A/B Testing
Correlation is not causation — and most business decisions require knowing causes, not correlations. This topic covers the statistical foundations of A/B testing, power analysis, the potential outcomes framework, observational causal methods (DiD, IV, matching), uplift modelling, and the DoWhy causal ML library. Senior DS roles in product and growth companies treat this as a core competency.
Why Causation Matters in ML
  • Correlation-based models fail under intervention: a model predicting retention based on login frequency will break if you force users to log in more — the feature changes but not through the causal pathway
  • Simpson's paradox: a drug appears to help in aggregate data but harms every subgroup — direction of effect reverses when confounders are controlled
  • Feedback loops: model predictions change user behaviour, which changes the training data — correlation breaks down
  • Policy decisions require causes: "should we increase X?" requires knowing whether X causes Y, not just that they correlate
The Potential Outcomes Framework (Rubin Causal Model)
$$\begin{aligned} Y_i(1) &= \text{outcome for unit } i \text{ if treated} \\ Y_i(0) &= \text{outcome for unit } i \text{ if not treated} \\ \text{ITE}_i &= Y_i(1) - Y_i(0) \quad \text{(Individual Treatment Effect — never observable)} \\ \text{ATE} &= \mathbb{E}[Y(1) - Y(0)] \quad \text{(Average Treatment Effect)} \\ \text{ATT} &= \mathbb{E}[Y(1) - Y(0) | T=1] \quad \text{(Effect on the Treated)} \end{aligned}$$
Fundamental problem of causal inference

We can never observe both $Y_i(1)$ and $Y_i(0)$ for the same unit at the same time. We observe either the treated or the control outcome — never both. All causal methods are strategies for estimating the counterfactual we cannot observe.

Part 1 — A/B Testing: The Gold Standard
Why Randomised Experiments Are the Gold Standard
Random assignment ensures treatment and control groups are identical in expectation on all covariates — observed and unobserved. This eliminates confounding: any difference in outcomes is caused by the treatment.
A/B test statistical setup
$$\begin{aligned} H_0&: \mu_{\text{control}} = \mu_{\text{treatment}} \quad (\text{no effect}) \\ H_1&: \mu_{\text{control}} \neq \mu_{\text{treatment}} \quad (\text{two-tailed}) \\ \text{Test stat}&: t = \frac{\bar{x}_T - \bar{x}_C}{\sqrt{s^2_T/n_T + s^2_C/n_C}} \\ p\text{-value}&: P(\text{data this extreme} | H_0 \text{ true}) \\ \text{Reject } H_0&\text{ if } p < \alpha \quad (\alpha = 0.05 \text{ typical}) \end{aligned}$$
Key concepts
  • Type I error (α): false positive — reject H₀ when it's true. α = 0.05 → 5% chance of false positive
  • Type II error (β): false negative — fail to reject H₀ when it's false
  • Statistical power (1-β): probability of detecting a true effect. Target ≥ 0.80
  • Minimum Detectable Effect (MDE): smallest effect size worth detecting; drives sample size
  • Two-tailed vs one-tailed: almost always use two-tailed — you can't know in advance which direction the effect will be
Power Analysis & Sample Size Calculation
$$n \approx \frac{2(z_{\alpha/2} + z_\beta)^2 \sigma^2}{\delta^2}$$
where $\delta$ = MDE (minimum detectable effect), $\sigma^2$ = variance of metric, $z_{\alpha/2}$ = 1.96 for α=0.05, $z_\beta$ = 0.84 for 80% power.
from scipy import stats
import numpy as np

def sample_size_per_group(
    baseline_rate=0.10,   # control conversion rate
    mde=0.02,             # min detectable effect (absolute)
    alpha=0.05,           # significance level
    power=0.80            # 1 - beta
):
    effect = mde
    pooled_p = baseline_rate + effect / 2
    pooled_var = 2 * pooled_p * (1 - pooled_p)
    z_alpha = stats.norm.ppf(1 - alpha / 2)
    z_beta  = stats.norm.ppf(power)
    n = (z_alpha + z_beta)**2 * pooled_var / effect**2
    return int(np.ceil(n))

# 10% baseline, want to detect +2pp lift, 80% power
n = sample_size_per_group(0.10, 0.02, 0.05, 0.80)
# n ≈ 3,842 per group (7,684 total)

Practical tip

Run your power analysis before launching the experiment — never stop early because it "looks significant." Peeking inflates Type I error. Pre-register the sample size and run to completion.

CUPED — the most impactful technique

CUPED uses the same metric measured before the experiment as a covariate to remove noise. If last week's revenue predicts this week's, regressing it out dramatically reduces variance. Used by Netflix, Airbnb, Booking.com as standard practice. Requires no change to randomisation — pure post-hoc analysis step.

Part 2 — Observational Causal Inference (When You Can't Randomise)
Difference-in-Differences (DiD)
$$\begin{aligned} \hat{\tau}_{\text{DiD}} &= (\bar{Y}_{T,\text{post}} - \bar{Y}_{T,\text{pre}}) - (\bar{Y}_{C,\text{post}} - \bar{Y}_{C,\text{pre}}) \\ &= \text{change in treated} - \text{change in control} \\ \text{Parallel trends assumption}&: \text{treated and control would have evolved similarly absent treatment} \end{aligned}$$
Classic example
  • Card & Krueger (1994): New Jersey raised minimum wage (treated); Pennsylvania did not (control)
  • Employment changed differently in NJ vs PA — the difference of differences is the causal effect
  • Key assumption: parallel trends — NJ and PA employment would have moved together absent the policy change
import statsmodels.formula.api as smf

# DiD regression formulation:
# Y = β0 + β1·post + β2·treated + β3·(post × treated) + ε
# β3 = DiD estimate (causal effect)

result = smf.ols(
    "outcome ~ post + treated + post:treated",
    data=df
).fit()

did_estimate = result.params["post:treated"]
print(f"DiD estimate: {did_estimate:.3f}")
print(f"p-value: {result.pvalues['post:treated']:.4f}")

Part 3 — Uplift Modelling (Heterogeneous Treatment Effects)
Uplift Modelling — Targeting Who Responds to Treatment
Average treatment effect tells you the mean effect. Uplift modelling estimates the individual treatment effect — who benefits most, who is unaffected, and who is harmed by the intervention.
The four customer segments
SegmentWithout treatmentWith treatmentTarget?
PersuadablesWon't convertWill convert✅ YES — highest ROI
Sure thingsWill convert anywayWill convert❌ Wasted spend
Lost causesWon't convertWon't convert❌ Wasted spend
Do-not-disturbWill convertWon't convert (harmed)🚫 AVOID
Methods
  • Two-model (T-learner): train separate models for treatment and control; uplift = model_T(x) - model_C(x). Simple but high variance
  • S-learner: single model with treatment indicator as feature; uplift = model(x, T=1) - model(x, T=0)
  • X-learner: cross-fitting that handles unequal treatment/control sizes; lower variance than T-learner
  • Causal forests (Wager & Athey): random forest variant that directly targets HTE estimation; gold standard in industry
  • Meta-learners (econML): plug-in estimation; works with any base learner
from econml.dml import CausalForestDML
from sklearn.ensemble import GradientBoostingRegressor, GradientBoostingClassifier

# Causal Forest: Y = outcome, T = treatment (binary), X = features, W = confounders
model = CausalForestDML(
    model_y=GradientBoostingRegressor(),
    model_t=GradientBoostingClassifier(),
    n_estimators=200, min_samples_leaf=5

model.fit(Y, T, X=X, W=W)

Part 4 — DoWhy & Causal Graphs
DoWhy — Microsoft's Causal Inference Library
DoWhy implements Pearl's causal framework: specify a causal graph (DAG), identify whether the causal effect is estimable from data, estimate it, and test robustness of the estimate.
import dowhy
from dowhy import CausalModel

model = CausalModel(
    data=df,
    treatment="ad_exposure",
    outcome="purchase",
    # Specify causal graph as GML or dot notation
    # Confounders: age, income affect both ad targeting AND purchase
    common_causes=["age", "income", "browsing_history"],
    instruments=["random_ad_slot"]  # instrumental variable
)

# Step 1: Identify — can we estimate the effect from observables?
identified_estimand = model.identify_effect(
    proceed_when_unidentifiable=True
)

# Step 2: Estimate using the identified strategy
estimate = model.estimate_effect(
    identified_estimand,
    method_name="backdoor.linear_regression"
    # also: "iv.instrumental_variable", "backdoor.propensity_score_matching"
)
print(f"Causal effect: {estimate.value:.4f}")

# Step 3: Refute — test sensitivity to assumptions
refute = model.refute_estimate(
    identified_estimand, estimate,
    method_name="random_common_cause"  # add random confounder; effect should not change
)
print(refute)
Causal graph concepts
  • DAG (Directed Acyclic Graph): nodes = variables, edges = direct causal relationships
  • Confounder: common cause of treatment and outcome — must be controlled
  • Collider: common effect of treatment and outcome — must NOT be controlled (conditioning opens a spurious path)
  • Backdoor criterion: a set of variables that blocks all backdoor paths from treatment to outcome and is not a descendant of treatment
  • Front-door criterion: estimate causal effect via mediators when confounders are unobserved

Common Mistakes & Misconceptions
  • Stopping the A/B test early because it looks significant — peeking inflates α. A test that hits p<0.05 at peek #5 has true α ≈ 0.19. Always pre-specify sample size and run to completion, or use sequential testing methods
  • Confusing statistical significance with practical significance — a 0.001% conversion rate lift can be statistically significant at n=10M but completely meaningless for the business. Always compute effect size and confidence intervals alongside p-values
  • Ignoring SUTVA violations in networked products — in social networks, marketplaces, or multiplayer games, treating one user affects others. Cluster randomisation or switchback designs are required
  • Controlling for colliders — adding a variable that is caused by both treatment and outcome opens a spurious backdoor path and introduces bias. Use a DAG to identify which variables to control and which to avoid
  • Confounding observational uplift estimates — self-selection means treated and untreated users differ systematically. Always use propensity score weighting or matching before computing uplift
  • Using A/B test results for causal claims without randomisation — quasi-experiments (DiD, RDD) require strong assumptions. Be explicit about what must be true for the estimate to be causal
How It's Used at Top Companies
  • Netflix: CUPED variance reduction as standard practice on all A/B tests; causal forests for personalised recommendation treatment effects
  • Airbnb: DiD for evaluating policy changes (e.g. host cancellation policy) where randomisation is infeasible; SUTVA-aware cluster experiments for marketplace interventions
  • Booking.com: runs 1,000+ concurrent A/B tests; CUPED + sequential testing to manage multiple comparisons and early stopping
  • Uber: switchback experiments for surge pricing (A/B by time period, not user, to avoid SUTVA violations in a marketplace)
  • Meta: uplift modelling for ad targeting — targeting persuadable users rather than "sure things" to reduce wasteful ad spend
  • Retail (Walmart, Target): synthetic control for evaluating store-level interventions (remodels, new product placement) where no randomisation is possible
🎯 Interview Gold

What's the difference between correlation and causation? "Correlation measures how two variables move together. Causation tells you that changing X produces a change in Y. Correlation can arise from confounding (a third variable causes both), reverse causation (Y causes X), or coincidence. The only way to establish causation is through randomisation (A/B test) or a valid natural experiment that mimics randomisation (DiD, IV, RDD)."

How do you design an A/B test? "Five steps: (1) Define the primary metric and MDE — the smallest lift worth detecting. (2) Run a power analysis to compute required n per group. (3) Randomise users (or clusters for networked products). (4) Run to the pre-specified sample size — no peeking, no early stopping. (5) Analyse: check sample ratio mismatch first, then run two-sided t-test, report effect size and 95% CI not just p-value."

What is CUPED? "CUPED reduces the variance of A/B test estimates by regressing out variance explained by a pre-experiment covariate (usually the same metric from the week before the experiment). The adjusted metric Y* = Y - θ(X - X̄) has lower variance but the same expectation. This is equivalent to running a larger experiment — 50% variance reduction halves the required sample size."

When can't you run an A/B test? "When randomisation is infeasible — policy changes affecting everyone, historical analysis, marketplace interventions. Then I'd use: DiD if there's a control group and parallel trends holds, RDD if there's a threshold with arbitrary assignment near it, IV if there's a valid instrument, or synthetic control for a single treated unit. Each requires explicit assumptions that must be stated and tested."
